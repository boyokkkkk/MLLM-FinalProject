## 摘要

本项目围绕教育学习、学术研究及日常办公场景中基于复杂文档信息的多模态问答任务，设计并实现了一套以结构化解析、块级索引、视觉辅助生成为核心的多模态检索增强生成系统。项目的初始目标是验证“文档结构感知 + 多模态检索增强”能否有效缓解传统纯文本问答系统在复杂页面理解、长文档检索与证据定位方面的局限；而在具体实现与实验推进过程中，我们进一步发现，真正决定系统结论是否可信的关键，不仅在于模型或模块是否足够复杂，更在于评测对象、索引对象与任务定义三者是否严格对齐。

围绕这一认识，我们完成了三方面工作。其一，构建了从数据标准化、MinerU 页面解析、块级语义切分、轻量索引构建、后端 问答服务到 benchmark 评测的完整工程闭环。其二，对原始 `DocVQA val`数据集构建`unique-docpage-100` 子集，固定评测清单以及与该清单严格对齐的重建索引语料。其三，在修复评测协议后重新审视视觉信息在系统中的作用位置，证明视觉信号在当前项目条件下更适合作为生成阶段的门控辅助，而不宜默认前置为检索阶段的强融合信号。

实验结果表明：在原始 `val-100` 评测基准上，以视觉稠密融合为核心的历史最佳检索线（`exp5_densefusion_w045`）仅取得 `Hit@5=0.51`、`Citation@1=0.13`；而在修复后的 `unique-docpage-100` 页级评测基准及其对应的重建索引语料上，以文本优先检索、题型感知重排和 query-image-aware rerank 为核心的检索主线（`stronger_qia_v6`）达到 `Hit@5=0.95`、`Precision@5=0.196`、`Citation@1=0.91`，进一步加入门控视觉辅助后的端到端主线（`stronger_qia_visualassist_gated_v2`）达到 `Hit@5=0.95`、`Citation@1=0.90`、`EM=0.87`、`ANLS=0.962800`、`Token-F1=0.951780`。研究结果说明，在课程规模的多模态文档问答系统中，稳定的文本优先检索、合理的页级评测构造以及题型敏感的视觉辅助机制，比单纯堆叠更多多模态融合模块，更能带来可信、可解释且可复现的系统提升。

## 1 引言

### 1.1 研究背景

在教学、科研与办公场景中，PDF 讲义、PPT 截图、扫描页、图表页、表单页等文档广泛存在。此类文档往往同时包含文本、表格、图像、公式、标题层级与版面布局等多种信息，其语义不仅来自文字内容本身，也来自页面结构、局部视觉线索与阅读顺序。用户在复习、问答、检索和跨页理解时，通常希望系统能够快速定位与问题对应的证据页面、证据区域与关键短语，而非仅返回模糊的“相关文本”。

传统基于纯文本切分的问答系统在此场景中存在天然不足。首先，纯文本抽取会丢失图表、标题、徽标、页码、表格边界等非线性语义信号，使得页面在被“线性化”后发生信息坍缩。其次，长文档往往超出单次上下文窗口，若检索机制缺乏对结构与题型的感知，生成模型即便具备较强语言能力，也难以稳定地回到真正相关的页面与区域。再次，文档问答不仅要求“答对”，还要求“答得可追溯”，即系统应尽量返回可验证的 citation，而非不可解释的自由生成结果。

随着多模态大模型与检索增强生成技术的发展，文档问答逐步从“整页 OCR 后交给语言模型”转向“结构化解析 + 块级检索 + 引用式生成”的体系化路线。然而，现有方法在实际工程落地时仍面临两个关键问题：一是视觉信息究竟应主要作用于检索阶段还是生成阶段；二是实验结论是否建立在与任务目标一致的评测协议之上。本文的结题工作，正是在这一背景下展开。

### 1.2 研究目标与核心任务

本研究旨在构建一个面向课程文档的多模态智能问答系统，实现对复杂页面的结构化解析、块级证据检索、可追溯回答生成与标准化实验评测。结合开题阶段的设想与后续实现演化，本文的核心任务可概括为以下四项：

1. 构建文档结构化解析与块级语义表示流程。围绕 `scripts/01_prepare_datasets.py`、`scripts/06_run_mineru.py` 与 `scripts/07_parse_and_chunk.py`三份数据集处理脚本，将原始 DocVQA/ChartQA 数据与用户文档统一转换为块级语义对象。
2. 建立统一的索引与检索主线。围绕 `scripts/08_build_indexes.py` 与 `src/models/retrieval.py`，实现以文本优先为主、视觉元数据为辅的证据召回与重排机制。
3. 构建从检索到回答再到引用返回的服务闭环。围绕 `src/serving/api.py`，打通 `/api/v1/chat` 的 `query -> retrieve -> generate -> citation` 主流程。
4. 建立可复现的 benchmark 评测与误差分析框架，形成实验记录、指标输出与问题诊断闭环。

### 1.3 研究问题

在系统实现与实验推进过程中，本文将项目目标进一步凝练为三个研究问题：

1. 原始 `DocVQA val first-100` 上的长期平台期，主要源自模型架构与检索能力不足，还是源自 benchmark 的任务定义偏差。
2. 在当前课程项目的数据规模与系统架构下，视觉信息最有效的接入位置究竟是检索阶段，还是生成阶段。
3. 在重建评测数据集修复 benchmark 之后，系统剩余错误的主要来源，究竟是普通排序噪声，还是更深层的 query grounding、不充分 OCR 表示与版面理解缺口。

### 1.4 研究推进主线

项目初期，我们主要关注基于开题报告的主线工程开发任务，将主要矛盾理解为“纯文本方法缺少视觉信息，因此需要更强的多模态融合”；中期，当系统主链路和初步 benchmark 建立后，我们观察到性能提升并不稳定，且不同实验线之间的差异难以形成可信解释；后期，通过误差分析与 benchmark 审视，我们意识到问题的核心并不首先是“模态不够多”，而是“评测对象、索引对象与证据定义并未充分对齐”。

基于这一认识，本文的叙事主线将组织为五个阶段：第一，完成从解析、检索到回答与 citation 返回的主线工程任务链；第二，在旧 benchmark 上识别异常现象进行分析诊断；第三，通过评测数据集错误采样暴露的重复页污染问题重新界定主要研究问题；第四，在修复 benchmark 后重新比较文本优先路线与视觉路线的真实收益；第五，将视觉信息从检索期强融合重定位为生成期门控辅助。换言之，希望本文能够真正解释清楚我们在实验过程中遇到的问题以及研究思路，进一步说明为什么系统会从最初设想演化到现在的主线形态，以及这种演化如何被实验事实所支持。

## 2 相关工作

### 2.1 多模态大模型与文档理解

多模态大模型为文档理解提供了新的技术基础。与只处理线性文本的语言模型相比，视觉语言模型能够同时处理页面图像与文字内容，因此在表单识别、图文推理、图表问答和页面短文本辨识等任务上具有明显优势。对于文档场景而言，这种优势尤其体现在标题、徽标、局部短语、页码与图表区域的识别上。

然而，多模态能力并不自动等价于可用的文档检索系统。若直接将整页乃至整本文档输入视觉语言模型，一方面会受到上下文长度与推理成本约束，另一方面也无法自然地解决“先找哪一页、再读哪一块”的证据组织问题。因此，多模态模型更适合作为文档问答链条中的推理器或局部视觉补偿器，而非唯一的信息组织机制。

### 2.2 文档 RAG 与多模态 RAG

RAG 通过“外部检索 + 条件生成”缓解了纯生成模型在长文档和外部知识利用上的限制，已经成为文档问答的重要技术路线。传统文本 RAG 的优点在于检索稳定、成本可控、可解释性较强，但其索引对象通常是线性文本片段，难以显式保留页面结构、图表证据与局部视觉信息。

近年来，多模态 RAG 开始将视觉路径引入检索与生成。VisDoM 强调视觉丰富文档中的多源召回与后融合，VisRAG 强调基于页面图像的检索价值，M3DocRAG 则进一步关注多页、多文档场景下的多模态证据组织。这些工作启发本文认识到：文档问答系统不应简单复制纯文本 RAG，也不应完全依赖整页视觉推理，而应在检索粒度、语义保持与可追溯性之间寻找工程上更稳健的平衡点。

### 2.3 结构感知切分与块级证据表示

除模型层面外，文档问答中的另一个关键问题在于切分。若切分方式不能保留版面结构与阅读顺序，即便检索命中了“相关文本”，也未必命中了“完整证据”。LayoutLMv3 等工作表明，文档的二维结构信息对语义理解具有重要作用；MinerU 等解析工具则从工程实践层面为标题、表格、图像与区域块的统一表示提供了可操作基础。

本文在实现中采用“块优先”思路，将 MinerU 输出视为主索引对象，并在 `chunks.jsonl` 中保留 `chunk_id`、`block_id`、`page_no`、`bbox`、`source_ref`、`source_path`、`image_path` 等回溯字段。这样的设计并不只是“切得更细”，而是为后续检索、重排、引用返回与误差分析提供了统一证据载体。

### 2.4 本文定位

综合上述工作可以看到，现有研究往往默认多模态信息越早介入越好，但在高度模板化、页面相似度高、重复视觉元素较多的课程文档场景中，这一前提未必成立。此外，若 benchmark 的构造方式与真实任务目标不一致，则实验结论也可能发生系统性偏移。

因此，本文将基于现有工作构建一条更可诊断、可验证、可复现实验结论的系统路线：以块优先结构为基础，以文本优先检索为主路径，以门控视觉辅助为补充路径，并通过 benchmark 修复保证结论的有效性。

## 3 系统设计与实现

### 3.1 总体设计思路

针对开题报告中提出的“语义丢失”与“长文档检索困难”两类核心问题，本文最终采用“结构化解析 + 块级表示 + 文本优先检索 + 视觉辅助生成”的总体路线。与开题阶段偏重“双路径 RAG + 晚融合”的设想相比，结题阶段最大的变化不在于模块数量的增减，而在于系统内部主次关系的重新排序：文本检索被确立为主证据排序机制，视觉信息则更多承担局部消歧与回答补偿的角色。

系统整体可划分为四个阶段：

1. 数据标准化与页面解析。
2. 块级语义对象构建与索引生成。
3. 证据召回、重排与组织。
4. 回答生成、citation 返回与离线评测。

这一设计的核心原则是：任何生成结果都应尽量能够回到具体页面与具体块，任何实验结论都应尽量能够回到明确的索引对象与评测协议。

### 3.2 数据标准化、MinerU 解析与块优先结构

数据入口由 `scripts/01_prepare_datasets.py`、`scripts/06_run_mineru.py` 与 `scripts/07_parse_and_chunk.py` 共同构成。前者负责将 DocVQA、ChartQA 等评测集统一标准化为 JSONL；然后将页面图片或 PDF 输入交给 MinerU，得到归一化的页面块；后者负责将解析结果转换为 `documents.jsonl` 与 `chunks.jsonl` 两类产物。

其中，`documents.jsonl` 主要记录文档级与样本级元信息，不直接作为检索对象；真正进入索引的是 `chunks.jsonl` 中的块级对象。每个块至少保留以下字段：`chunk_id`、`document_id`、`block_id`、`page_no`、`bbox`、`source_ref`、`source_path`、`image_path` 与 `metadata`。项目在后期放弃将 benchmark 问题文本作为默认检索对象的做法，仅将其保留为早期 smoke test 所需的可选调试机制。这一转变意味着系统从记住问题走向读取页面，从而保证检索对象与真实任务对象保持一致。

为便于明确文档级对象与块级对象的职责边界，本文将两类核心数据结构概括如表 1 所示。

| 对象              | 角色定位     | 核心字段                                                                                                                                            | 是否直接进入索引系统 | 主要用途                                              |
| ----------------- | ------------ | --------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------- | ----------------------------------------------------- |
| `documents.jsonl` | 文档级父对象 | `document_id`, `dataset`, `split`, `source_type`, `source_path`, `image_path`, `page_no/page_count`, `question`, `answers`, `metadata`              | 否                   | 保存文档来源、评测样本信息与父级回溯信息              |
| `chunks.jsonl`    | 块级证据对象 | `chunk_id`, `document_id`, `block_id`, `block_index`, `part_index`, `chunk_type`, `text`, `page_no`, `bbox`, `source_ref`, `image_path`, `metadata` | 是                   | 作为检索、重排、citation 返回与视觉辅助的统一证据单元 |

表 1 表明，本文并不把文档级记录与检索级记录混为一谈。`documents.jsonl` 负责回答“这是什么文档、来自哪里、对应什么评测样本”，而 `chunks.jsonl` 负责回答“哪一段证据应进入索引、应如何被召回、以及应如何回溯到原页面区域”。

上述数据构建过程可抽象为算法 1。

```text
算法 1 文档解析与块级切分流程
输入：原始输入 x（PDF、页面图像或标准化评测样本）
输出：文档级记录 d，与块级证据集合 C

1: 若 x 来自公开数据集，则先标准化为统一评测样本格式
2: 提取 x 对应的页面图像或 PDF 路径
3: 调用 MinerU 对页面进行结构化解析，得到语义块集合 B
4: 构建文档级记录 d，写入 document_id、source_path、image_path、page 信息与 metadata
5: 初始化块级集合 C ← ∅
6: 对每个语义块 b ∈ B 执行：
7:     规范化块类型、页码、bbox、image_path 与原始解析字段
8:     若 b 的文本长度 ≤ max_chars_per_chunk，则生成一个 chunk c
9:     否则在 b 内部按滑窗切分，生成多个 part，并共享同一 block_id
10:    若 b 为 figure/table/formula/page_image 且文本为空，则补充占位文本
11:    为每个 c 写入 chunk_id、document_id、block_id、page_no、bbox、source_ref、metadata
12:    将 c 加入 C
13: 返回 d 与 C
```

### 3.3 索引构建与文本优先检索主线

`scripts/08_build_indexes.py` 负责基于块级对象构建本地轻量索引。当前主线索引主要包含三类对象：一是以 `doc_store.json`、`postings.json` 与 `document_frequency.json` 为核心的稀疏文本索引；二是 `visual_store.json` 形式的视觉元数据索引；三是部分实验中使用的视觉描述符存储与向量文件。

核心检索逻辑位于 `src/models/retrieval.py`。从机制上看，系统采用“两阶段检索”思路：第一阶段先用轻量的文本相似度方法，从全部块级证据中快速筛出一小批候选；第二阶段再对这些候选进行更细致的重排判断。这样设计的原因在于，文档问答并不只是“找词最像的文本”，而是要在大量页面块中找出“最可能真正回答当前问题的证据块”。

在重排阶段，系统不再把所有问题都当作同一类文本匹配任务处理，而是显式区分不同题型对证据形态的依赖差异。对于标题、页眉、徽标、页码等问题，系统会更偏好短标题块、页面顶部块或具有明显版面标记的块；对于表格与图表问题，系统会更重视包含结构化数值信息的文本块；对于明显缺乏信息量的通用占位块、噪声块或与问题仅存在表面重合的块，则给予额外惩罚。换言之，系统试图学习的不是“哪段话和问题最像”，而是“哪类证据在当前题型下最有可能是正确证据”。

后期优化中的一个关键改动，是在更强检索配置下，先削弱问题中低信息词对初筛结果的影响，再进行候选排序。这样做的背景是：我们在误差分析中发现，一些由调试机制生成的 synthetic chunk 会重复题干表述，因此很容易仅凭表面词汇重叠获得较高分数，进而压制真正来自页面内容的证据块。通过降低这类低信息词和题干回声的支配作用，系统能够把注意力更多放在实体词、结构词和页面内容本身上。其结果并不是简单地“多加几条规则”，而是让检索过程更接近文档问答任务真正关心的目标，即从原始页面中定位可信、可回溯且与当前问题语义对应的证据。

对应的主检索流程可概括为算法 2。

```text
算法 2 文本优先检索与重排流程
输入：用户问题 q，索引语料 I，返回条数 k
输出：排序后的证据列表 E

1: 对 q 进行分词，得到 query tokens
2: 若 rerank_profile = stronger，则过滤低信息 stopwords
3: 基于稀疏表示在 I 中计算 chunk 级初始相似度分数
4: 选取前 N 个候选块构成候选池 P
5: 对每个候选块 p ∈ P 计算重排增益：
6:     判断 q 是否属于 title / heading / logo / page number / table / chart 等题型
7:     对短标题块、页码样式块、结构化表格文本给予奖励
8:     对低信息噪声块、通用占位块或错误词面回声块给予惩罚
9:     若启用 query-image-aware rerank，则对 deictic/layout 问题追加页面级辅助重排
10: 合并基础分数与重排增益，得到最终分数
11: 按最终分数降序排序，并截取 top-k 结果
12: 返回 E
```

### 3.4 服务接口、证据组织与引用返回

`src/serving/api.py` 提供完整的后端服务闭环。系统通过 `/api/v1/chat` 接收 query、上下文与可选图片输入，调用 `LocalTextRetriever` 召回证据块，再将证据组织为生成提示词，最后由视觉语言模型生成回答并返回 citation。citation 至少包含 `chunk_id`、`source`、`page`、`snippet` 等字段，并通过 `source_ref` 保留到原始页面与区域的回溯路径。

### 3.5 从检索期视觉融合到生成期门控视觉辅助

项目关于视觉路径的实现经历了明显演化。早期思路倾向于在检索阶段直接进行视觉稠密融合，希望利用页面图像的全局相似性提升召回质量；相应地，`src/evaluation/benchmark.py` 中也实现了视觉融合与 query-image-aware rerank 等实验路径。然而，后续实验表明，在页面布局相近、标题模板重复、通用视觉元素较多的课程文档场景中，视觉描述符容易把“看起来相似”但证据并不正确的页面推到更前位置。

基于这一观察，项目后期将视觉信息更多转移到生成阶段使用，实现了门控视觉辅助策略：仅在 `logo`、`pack text`、`heading`、`title`、`page number`与短手写文本等题型上启用视觉辅助；对于`chart`、`table`、一般数字题与抽象语义题则禁用该机制。该策略保留了视觉信息对局部短文本辨识的价值，同时避免其在不适合的题型上扰动文本证据主线。

门控视觉辅助的触发逻辑如算法 3 所示。

```text
算法 3 门控视觉辅助触发逻辑
输入：用户问题 q，顶部证据列表 E，视觉辅助策略 policy
输出：是否启用视觉辅助的布尔值 flag

1: 对 policy 做归一化
2: 若 policy = off，则返回 False
3: 若 policy = always，则返回 True
4: 对 q 做小写化与题型词检测
5: 定义启用标记：
6:     logo / brand / pack text / heading / title / page number / handwritten
7: 定义禁用标记：
8:     chart / table / axis / amount / value / summary / abstract 等
9: 若 q 命中禁用标记，则返回 False
10: 若 q 命中启用标记，则返回 True
11: 若 q 较短，且顶部证据中存在 page_image 或 figure 类型证据，则返回 True
12: 否则返回 False
```

## 4 实验设计

### 4.1 数据集与评测对象

本文主要围绕 DocVQA 进行系统性实验，并辅以 ChartQA 的流程验证。

其中，原始历史基准使用 `DocVQA val first-100`；后期主基准使用 `unique-docpage-100`。后者通过 `scripts/15_build_unique_docpage_benchmark.py` 从 `data/processed/docvqa/val.jsonl` 出发，按 `(ucsf_document_id, ucsf_document_page_no)` 分组，保留每个文档页一个代表性问题，并结合 question type 做分层抽样，最终得到固定的 100 页 manifest。同步重建了 `documents`、`chunks` 与 `indexes`，从而形成与 manifest 严格对齐的 rebuilt corpus。

### 4.2 指标体系

本文采用的指标体系包括三类：

1. 检索指标：`Hit@5` 与 `Precision@5`。其中，`Hit@5` 表示在系统返回的前 5 个证据块中，是否至少出现了一个与标准答案对应的正确证据；该指标主要衡量系统能否把正确页面或正确证据召回到候选集合中。`Precision@5` 表示前 5 个返回结果中相关证据所占的比例；该指标不仅关心“有没有找对”，也关心“候选列表中有多少噪声”，因此更能反映检索排序的纯度。
2. 引用指标：`Citation@1`。该指标考察系统返回的第 1 条 citation 是否指向正确证据来源。在本文场景下，它比一般召回指标更严格，因为它直接对应用户最终最先看到的证据是否可信，因而可以近似衡量系统的 top-1 证据定位能力。
3. 回答指标：`Exact Match`、`ANLS`、`Token-F1`，以及部分分析中使用的 `Answer-Contains`。其中，`Exact Match` 要求模型输出与标准答案在归一化后完全一致，适合衡量答案是否“严格答对”；`ANLS`（Approximate Normalized Levenshtein Similarity）允许小幅字符差异，尤其适合 OCR 噪声、大小写差异或轻微拼写偏差较常见的文档问答场景；`Token-F1` 从词项重合角度综合考虑准确率与召回率，能够反映预测答案与标准答案在内容层面的整体接近程度；`Answer-Contains` 则用于补充分析，主要判断系统回答中是否至少包含标准答案的关键信息，适合观察回答虽然不完全规范、但是否已经抓住核心内容。

此外，`src/evaluation/benchmark.py` 中同时保留了 sample-level 与 doc-page-level 两种相关性判定逻辑。前者强调是否命中完全一致的样本来源，后者强调是否命中同一原始文档页。

### 4.3 对比实验设置

为分析系统演化中的真实收益，本文重点比较以下代表性实验线：

1. `promptfix_v3`：旧 benchmark 上较稳定的文本基线。
2. `exp5_densefusion_w045`：旧 benchmark 上表现最好的视觉稠密融合线。
3. `stronger_rebuilt`：修复 benchmark 后的文本优先重排主线。
4. `stronger_qia_v6`：加入 query-image-aware rerank 后的当前最佳检索线。
5. `stronger_qia_base`：修复 benchmark 后的 clean grounding-first RAG 基线。
6. `stronger_qia_visualassist_gated_v2`：当前最佳端到端主线。

## 5 实验过程中的关键问题与解决思路

### 5.1 原始 benchmark 的重复页污染问题

在对 `outputs/eval/docvqa_val_100_error_analysis_exp5.md` 进行误差分析后，项目发现原始 `val-100` 中虽然包含 100 个 QA 样本，但实际上只对应 29 个唯一文档页。这意味着许多不同 sample 共享同一原始页面，而 sample-level 指标会把“命中同页但命中到其他 sample 副本”的情况记为错误。

若继续在旧 benchmark 上堆叠视觉模块，已无法回答“系统究竟没有找到正确页面，还是只没有找到当前 sample 的那一份副本”。因此，项目先转向“如何首先修复 benchmark 的任务定义偏差”，重新构建了一版测试数据集。关于数据集的数量选择，是由于小组成员的算力有限，我们目前选择采样100条docvqa数据来作为后续所有测试实验的数据支撑。

### 5.2 固定 manifest 与 rebuilt corpus 的设计

针对重复页污染问题，项目新增 `scripts/15_build_unique_docpage_benchmark.py`处理脚本，以 `(ucsf_document_id, ucsf_document_page_no)` 作为分组键，仅保留每页一个代表性问题，并通过题型分层采样构造 `unique-docpage-100` 固定评测清单。随后又进一步重建与该清单对齐的 `documents`、`chunks` 与 `indexes`，避免出现评测切片变了，但检索语料仍保留旧分布的结构性偏差。

在此之前，项目面对的是某些实验线为什么提不上去的局部优化问题；解决了数据集层面的问题，后续关于文本主线、视觉辅助与剩余错误来源的讨论才有了可靠前提。

### 5.3 `fallback_text` 对 top-1 的劫持问题

在评测基准修复后，系统性能虽然显著提升，但 top-1 仍存在一批错误。进一步诊断表明，这些错误并不主要意味着“真实 OCR 块已经被召回，只是排序稍有偏差”，而是说明在很多样本里，排序最靠前的位置首先被一类 synthetic `fallback_text` 块占据了。所谓 `fallback_text`，本质上是早期调试或兜底机制产生的辅助文本块，这类块往往直接复述问题本身，或保留与题干高度相似的模板化表述。因此，在优化前的 top-1 结果中，`99/100` 来自 `fallback_text`，且 `99/100` 来自 `parser=fallback`，只有 `1/100` 来自真实 MinerU OCR 块。

这一现象说明，当索引系统同时混有“真实页面证据块”和“问题回声块”时，稀疏检索会被表面词汇重叠显著误导。对于初始候选筛选而言，一个反复复述题干关键词的辅助块，往往比真实页面中的自然文本看起来“更像问题”，即使后者才是真正可用于回答问题的证据来源。于是，系统表面上仍在执行文档检索，实际上却优先匹配了与问题最相似的人工回声文本。这会带来两个后果：一方面，真实页面内容的排序能力被掩盖；另一方面，后续优化也容易误以为主要问题来自 OCR 质量或细粒度重排本身。

针对这一问题，项目在 `src/models/retrieval.py` 中对更强检索配置做出调整：在稀疏预排序阶段，先削弱低信息词和整句题干回声对初始相似度的影响，再进入后续候选排序。这样做的目标，不是简单删除某些词，而是降低“问题模板复述”对排序结果的支配作用，让真正包含页面内容、实体信息和结构信号的证据块更容易进入候选前列。该改动使 `Hit@5` 从 `0.93` 提升到 `0.95`，`Precision@5` 从 `0.19` 提升到 `0.196`，`Citation@1` 从 `0.89` 提升到 `0.90`。更重要的是，它把优化重点重新拉回到了“真实页面证据的排序质量”本身，使后续分析能够更准确地区分：究竟是页面内容没有被召回，还是召回后没有被正确排序。

### 5.4 视觉稠密融合的负结果

在原始 benchmark 上，`exp5_densefusion_w045` 曾将 `Citation@1` 从 `0.12` 提升到 `0.13`，因此一度被视为较有前景的方向。然而，在 `unique-docpage-100 rebuilt` 上，对应的 `exp5_rebuilt` 并未超过文本优先主线，反而在 top-1 citation 上略弱于 `stronger_qia_v6`。

它说明视觉稠密融合并非天然优于文本优先检索。对于布局相近、模板重复、页面视觉元素高度相似的文档，视觉描述符容易把“相似页面”提前，而不一定把“正确证据页面”提前。因此，“把视觉信息尽早前置到检索阶段”在当前项目条件下并不是稳妥策略。

### 5.5 生成阶段视觉辅助的门控优化

当检索阶段视觉融合未能成为主线后，项目转向生成阶段视觉辅助。初版全局视觉辅助虽然在个别样本上改善了 `logo` 与 `title` 类问题，但整体 `ANLS` 与 `Token-F1` 出现下降，说明视觉信息并非越多越好，若缺乏题型约束，反而可能干扰稳定的文本 grounding。

基于 `outputs/eval/docvqa_val_unique_docpage_100_visual_assist_experiment.md` 的对比记录，项目最终引入门控策略，仅对 `logo`、`pack text`、`heading`、`title`、`page number` 与短手写文本问题启用视觉辅助，并对 `chart axis`、数字表格题和一般抽象语义题禁用该机制。该策略最终形成 `stronger_qia_visualassist_gated_v2`，并成为当前最稳健的端到端路线。这一结果说明，视觉信息的最佳系统角色不是“无差别增强”，而是“在文本证据已稳定时，对特定局部识别任务进行定向补偿”。

本文并未简单否定视觉信息的价值，而是通过一系列负结果与对照实验，完成了对视觉信息“系统角色”的重新界定。换言之，项目后期的贡献不是证明“视觉无用”，而是证明“视觉何时有用、为何在此时有用、以及为何不应在彼时强行介入”。

## 6 实验结果与分析

本节不再按实验日志的先后顺序堆叠结果，而是围绕一个更清晰的问题组织：究竟是哪个模块，带来了什么样的提升，或引入了什么样的负面效果。为避免不同任务口径混杂，本文将结果分为三层解释：第一层是评测基准修复带来的“测量校正”；第二层是检索链路内部各模块带来的“检索质量变化”；第三层是视觉模块在不同接入位置上带来的“回答质量变化”。

### 6.1 评测基准修复

原始 `DocVQA val first-100` 上的历史结果如下表所示。

| line                    | 概述             | type      | Hit@5 | Precision@5 | Citation@1 |   EM |     ANLS | Token-F1 |
| ----------------------- | ---------------- | --------- | ----: | ----------: | ---------: | ---: | -------: | -------: |
| `promptfix_v3`          | 旧文本检索基线   | retrieval |  0.49 |       0.114 |       0.12 |    - |        - |        - |
| `exp5_densefusion_w045` | 旧视觉稠密融合线 | retrieval |  0.51 |       0.118 |       0.13 |    - |        - |        - |
| `promptfix_v3`          | 旧文本端到端基线 | rag       |  0.49 |       0.114 |       0.12 | 0.78 | 0.874770 | 0.897306 |
| `exp5_densefusion_w045` | 旧视觉端到端主线 | rag       |  0.51 |       0.118 |       0.13 | 0.77 | 0.878745 | 0.898544 |

若只看这些数字，很容易得出“视觉稠密融合略优于文本基线”的印象；但这一结论并不可靠，因为旧基准本身混入了大量重复页，导致 sample-level 打分无法真实反映系统是否已经找到了正确文档页。换言之，这一阶段结果的主要价值，不是证明某条方法线更强，而是暴露出评测协议本身存在偏差。

在修复为 `unique-docpage-100` 并重建对应索引语料之后，检索结果发生了结构性变化，如表 5 所示。

| line               | 概述                       | Hit@5 | Precision@5 | Citation@1 |
| ------------------ | -------------------------- | ----: | ----------: | ---------: |
| `stronger_rebuilt` | 修复基准后的文本优先主线   |  0.93 |        0.19 |       0.89 |
| `stronger_qia_v6`  | 加入图像感知重排的检索主线 |  0.95 |       0.196 |       0.91 |
| `exp5_rebuilt`     | 修复基准后的视觉融合线     |  0.93 |        0.19 |       0.88 |

基准修复后，文本优先主线反而优于此前看似更有前景的视觉稠密融合线，因而后文关于各模块贡献的分析，都以修复后的 `unique-docpage-100` 为准。

![图 1 基准修复对评测口径的影响](/D:/MLLMProject/outputs/eval/final_assets/fig_benchmark_repair.svg)

图 1 展示了原始 `val-100` 与修复后 `unique-docpage-100` 在页级唯一性上的根本差异。它对应本节最重要的结论：benchmark 修复首先改变的是“系统被如何测量”，然后才谈得上“系统被如何比较”。

### 6.2 文本优先排序

在统一基准上，索引表示方式的差异并未成为主要决定因素，如表 6 所示。

| index variant      | 概述           | Hit@5 | Citation@1 | DocPage Hit@5 |
| ------------------ | -------------- | ----: | ---------: | ------------: |
| `page_text`        | 页级文本索引   |  0.95 |       0.91 |          0.95 |
| `block_text`       | 块级文本索引   |  0.95 |       0.91 |          0.95 |
| `block_multimodal` | 块级多模态索引 |  0.95 |       0.91 |          0.95 |

这说明在当前 100 页评测子集上，系统性能并不是靠“把索引切得更细”或“把索引改成多模态表示”获得的。页级文本、块级文本和块级多模态索引表现几乎一致，表明当前最关键的问题并不是索引颗粒度，而是检索排序本身是否能够把正确证据放到前列。

进一步看检索增强链路的消融，如表 7 所示。

| retrieval line   | 概述                       | Hit@5 | Citation@1 | DocPage Hit@5 |
| ---------------- | -------------------------- | ----: | ---------: | ------------: |
| `basic`          | 基础文本检索               |  0.93 |       0.89 |          0.93 |
| `stronger`       | 强化文本重排               |  0.95 |       0.90 |          0.95 |
| `stronger + qta` | 强化文本重排加题型感知重排 |  0.95 |       0.90 |          0.95 |
| `stronger + qia` | 强化文本重排加图像感知重排 |  0.95 |       0.91 |          0.95 |

从这组结果可以把不同模块的作用拆开来看。`basic -> stronger` 是检索链路中最主要的一步提升：`Hit@5` 从 `0.93` 提高到 `0.95`，`Citation@1` 从 `0.89` 提高到 `0.90`。这一提升主要来自更稳健的文本优先排序，包括削弱低信息词干扰、抑制问题回声块、以及改进候选重排逻辑，把排序对象重新拉回到“真实页面证据”。

相比之下，`query-type-aware rerank` 在当前子集上的额外增益较小，说明题型启发式在这一固定样本集上更多扮演“稳定器”而不是“主要增长点”。而 `query-image-aware rerank` 虽未继续提高 `Hit@5`，却把 `Citation@1` 从 `0.90` 推高到 `0.91`，说明它更主要地改善了 top-1 证据排序，而不是扩大候选召回范围。也就是说，检索链路中的核心增益来自文本优先主线本身，后续视觉感知重排只提供了相对有限但方向明确的补充。

![图 2 修复基准上的检索链路消融](/D:/MLLMProject/outputs/eval/final_assets/fig_retrieval_progression.svg)

图 2 对应表 6 与表 7，可直观看到 `basic -> stronger` 是检索链路中的主要跃迁，而 `stronger + qia` 的额外收益更集中体现在 top-1 grounding，而不是整体召回。

### 6.3 视觉信息前置到检索阶段

为了检验“更强多模态检索”是否真的优于文本优先主线，本文进一步比较了文本优先主线与 retrieval-time dense fusion，如表 8 所示。

| line                 | 概述                       | Hit@5 | Citation@1 | DocPage Hit@5 |
| -------------------- | -------------------------- | ----: | ---------: | ------------: |
| `stronger + qia`     | 文本优先加图像感知重排主线 |  0.95 |       0.91 |          0.95 |
| `visual_densefusion` | 检索阶段视觉稠密融合       |  0.95 |       0.79 |          0.95 |

这是一组非常关键的负结果。视觉稠密融合并没有降低页级召回，`Hit@5` 仍保持 `0.95`；但它显著破坏了 top-1 证据排序，使 `Citation@1` 从 `0.91` 下降到 `0.79`。这说明视觉信息在当前课程文档场景下并非“越早介入越好”。对于布局近似、模板重复、视觉元素高度相似的页面，视觉描述符更容易把“相似页”排到前面，而不一定把真正含有答案的证据页排到前面。

因此，视觉模块在检索阶段带来的并不是稳定增益，而是一个明确的风险：它可能保住了“找对页”的能力，却损害了“把最正确的证据放在第一位”的能力。对以 citation 为核心的文档问答系统而言，这种负面效果比单纯召回下降更值得警惕。

### 6.4 视觉信息放到生成阶段后

在确认视觉稠密融合不适合作为检索主线后，项目将视觉信息转移到生成阶段，并区分两种评测口径。第一种是 `Corpus-grounded RAG`：模型不能直接看题图，必须依赖语料库中的页面证据作答；第二种是 `Query-image assisted QA`：允许模型直接观察 query page image，用于估计视觉识别上限。前者用于支持论文主结论，后者仅作补充诊断。

先看 `Corpus-grounded RAG` 的结果，如表 9 所示。

| line                     | 概述                   | Hit@5 | Citation@1 |   EM |     ANLS | Token-F1 |
| ------------------------ | ---------------------- | ----: | ---------: | ---: | -------: | -------: |
| `cg_text_only`           | 仅用语料文本证据作答   |  0.95 |       0.90 | 0.13 |     0.13 | 0.147952 |
| `cg_visualassist_always` | 始终启用视觉辅助作答   |  0.95 |       0.90 | 0.82 | 0.834982 | 0.839753 |
| `cg_visualassist_gated`  | 按门控策略启用视觉辅助 |  0.95 |       0.90 | 0.20 |     0.20 | 0.216080 |

这一组结果把视觉辅助的贡献解释得非常清楚。三条实验线的检索指标完全相同，说明差异不是由检索模块带来的，而是由生成阶段是否有效利用视觉证据决定的。`cg_text_only` 的 `EM` 只有 `0.13`，说明在不直接看题图、只依靠文本证据的情况下，模型很难稳定恢复短答案。相反，`cg_visualassist_always` 将 `EM` 提升到 `0.82`，同时显著提高 `ANLS` 与 `Token-F1`，表明视觉辅助对局部短文本、标题、页码和手写短语等问题具有决定性价值。

但这组结果也同时暴露出当前门控策略的不足。`cg_visualassist_gated` 虽优于 `cg_text_only`，却仅达到 `EM=0.20`，与 `always` 之间仍有巨大差距。这意味着问题不在于“视觉有没有用”，而在于“现有 gate 过于保守，没有把视觉能力在该用的时候及时调动起来”。

为进一步验证这一判断，表 10 对不同 gate 版本进行了细拆。

| gate policy              | 概述                         | Hit@5 | Citation@1 |   EM |     ANLS | Token-F1 |
| ------------------------ | ---------------------------- | ----: | ---------: | ---: | -------: | -------: |
| `cg_text_only`           | 仅用语料文本证据作答         |  0.95 |       0.90 | 0.13 |     0.13 | 0.147952 |
| `cg_logo_only`           | 仅在徽标题触发视觉辅助       |  0.95 |       0.90 | 0.16 |     0.16 | 0.176928 |
| `cg_title_page_only`     | 仅在标题与页码题触发视觉辅助 |  0.95 |       0.90 | 0.18 | 0.195627 | 0.206292 |
| `cg_handwritten_only`    | 仅在手写题触发视觉辅助       |  0.95 |       0.90 | 0.14 |     0.14 | 0.157448 |
| `cg_visualassist_gated`  | 统一门控视觉辅助             |  0.95 |       0.90 | 0.20 |     0.20 | 0.216080 |
| `cg_visualassist_always` | 始终启用视觉辅助             |  0.95 |       0.90 | 0.82 | 0.834982 | 0.839753 |

可以看到，无论是单独题型门控，还是统一启发式门控，都只带来了有限提升；真正的大幅改善来自 `always`。因此，生成阶段视觉辅助本身是有效模块，但当前“是否触发”的策略仍是系统最薄弱的部分。换言之，视觉模块的正贡献已经被实验充分证明，真正限制其收益释放的，是 gate 设计，而不是视觉信息本身。

图 3 将表 9 与表 10 的核心趋势压缩到了同一张图中：在检索指标几乎不变的前提下，回答质量差异主要由生成阶段是否成功利用视觉证据决定。

### 6.5 模型的视觉能力不是主要瓶颈

作为补充，`Query-image assisted QA` 的结果如表 11 所示。

| line                       | 概述                             |   EM |     ANLS | Token-F1 |
| -------------------------- | -------------------------------- | ---: | -------: | -------: |
| `qiaqa_text_only`          | 直接看题图作答文本基线           | 0.93 | 0.945627 | 0.944373 |
| `qiaqa_logo_only`          | 直接看题图且仅徽标题启用辅助     | 0.92 | 0.940788 | 0.938171 |
| `qiaqa_title_page_only`    | 直接看题图且仅标题页码题启用辅助 | 0.93 | 0.945627 | 0.945154 |
| `qiaqa_handwritten_only`   | 直接看题图且仅手写题启用辅助     | 0.93 | 0.945627 | 0.944989 |
| `qiaqa_visualassist_gated` | 直接看题图且采用统一门控辅助     | 0.93 | 0.945627 | 0.941824 |

当模型被允许直接观察题图时，各种设置几乎都稳定在 `EM≈0.93`。这说明模型并不缺少基本视觉识别能力，系统真正的瓶颈在于 corpus-grounded 条件下如何把视觉能力与语料库证据稳定连接起来。也正因为如此，本文不把这组结果当作主结论，而是把它作为一项诊断性证据：问题主要出在视觉证据的触发与组织，而不是模型“看不懂图”。

### 6.6 本章小结：各模块的真实贡献与代价

综合本章结果，可以将各模块作用概括为四点：

1. 评测基准修复带来的不是普通参数提升，而是对系统真实能力的重新测量；它决定了后续结论是否可信。
2. 检索链路中，最主要的正向贡献来自更稳健的文本优先排序；题型感知和 query-image-aware 重排只提供了有限但有方向性的补充。
3. 视觉信息若前置到检索阶段，会明显损害 top-1 citation 排序，因此不适合作为当前系统的默认主线。
4. 视觉信息若后置到生成阶段，则能显著提升回答质量；但当前门控策略过于保守，尚未充分释放这一模块的真实收益。

## 7 讨论

### 7.1 相比开题设想的真实变化

开题报告中的系统设想更接近“结构化切分 + 双路径 RAG + 晚融合”的典型多模态方案。结题阶段表面上仍保留了这一框架，但其内部逻辑已经发生重要变化：系统不再默认认为“多模态越早介入越好”，而是通过实验逐步建立了更细化的角色分配，即文本路径负责主证据排序，视觉路径负责局部识别补偿，benchmark 修复负责保障实验可解释性。

这种变化说明，项目的成熟并不只体现在功能变多或分数升高，更体现在系统主次关系的认识更加清晰。相比“证明某个模块有用”，本文更重要的贡献在于澄清“该模块在什么条件下、以什么方式有用”。

### 7.2 本项目最值得保留的研究经验

从研究方法角度看，本项目最值得保留的经验主要有三点。

第一，benchmark 并不是中性的。若评测对象与真实任务对象不一致，分数既可能夸大问题，也可能掩盖真实进步。第二，多模态信息不是天然正增益。其有效性取决于介入时机、题型条件与系统中的证据角色。第三，工程闭环与科学诊断同等重要。只有当 schema、API、citation、评测脚本与误差分析被统一到同一框架内，实验现象才真正具备可解释性。

如果进一步概括，这三条经验实际上共同指向一个更一般的研究原则：对于工程型多模态系统，真正高质量的结题工作不应仅展示“做成了什么”，更应解释“为何这样做、为何此前的做法不够好、以及结论为何值得相信”。本文的主要价值，正是在课程项目范围内尽可能完成了这一层解释任务。

## 8 局限性与后续工作

尽管当前系统已形成较完整的离线评测闭环，但仍存在以下局限。

1. 当前主要结论建立在 DocVQA 风格数据上，`unique-docpage-100` 虽比旧基准更合理，但仍属于固定切片，外部泛化能力仍需在更多文档域上验证。
2. 当前检索优化仍包含较强启发式成分，虽然在课程项目中有效，但未必能无缝推广到更大规模开放域语料。
3. 当前 benchmark 主要衡量页级检索与页级 grounding，而非更细粒度的区域级定位；页面正确不等于区域必然正确。
4. 在线文件上传、增量解析、实时建索引与长期 workspace 管理尚未成为正式实验对象，系统当前更接近研究原型而非完整产品。

据此，后续工作可沿三条路线展开：

1. 推进在线文档接入与增量索引，使系统从离线 benchmark 原型进一步走向真实交互应用。
2. 在现有 query-aware rerank 的基础上引入更学习化的页级或区域级 reranker，提升 generic/deictic query 的 grounding 能力。
3. 结合不确定性估计进一步细化视觉辅助门控策略，仅在模型对局部视觉短文本存在明显识别困难时触发视觉补偿。

## 9 结论

本文围绕课程文档场景下的多模态问答任务，完成了一套以块优先结构、可追溯引用与 benchmark 驱动诊断为核心的多模态 RAG 系统实现与实验分析。与开题阶段相比，结题阶段最重要的进展并不只是“做出了一个能运行的系统”，而是通过一系列失败分析、评测修复与实现重构，澄清了系统中真正的主要矛盾。

研究结果表明，原始 `DocVQA val first-100` 上看似持久的性能平台期，在很大程度上来自 benchmark 的重复页污染；在构造 `unique-docpage-100` 并重建对应索引语料后，文本优先检索能力得到了更真实的反映。同时，视觉信息虽然对页面理解不可或缺，但其最有效的系统角色并不是默认前置到检索排序，而是在文本证据已经稳定的前提下，以题型敏感的方式辅助生成模型完成局部视觉辨识。

因此，本文最终给出的判断：对于课程规模的多模态文档问答系统，评测对象与索引对象的一致性、文本优先的稳定检索主线，以及题型敏感的门控视觉辅助机制，比简单堆叠更多多模态融合模块，更能带来可信、可解释且可复现的系统提升。
