# 多模态文档问答系统实验报告

## 摘要

本文面向课程场景下的多模态文档问答任务，设计并实现了一套以结构化文档解析、块级检索、证据可追溯回答和分层评测为核心的多模态 RAG 系统。项目初期的目标是验证“文本检索 + 视觉增强”能否在 DocVQA 风格任务上优于纯文本基线；但在系统推进过程中，我们发现真正影响实验结论可信度的关键，并不只是模型结构本身，而是评测对象、索引对象与检索策略之间是否保持一致。围绕这一认识，项目逐步完成了三项核心工作：其一，建立从标准化数据、MinerU 解析、chunk 构建、索引生成到 API 推理的完整工程闭环；其二，通过误差分析发现原始 `DocVQA val first-100` 子集存在严重的重复页污染，并据此重建 `unique-docpage-100` 固定评测基准；其三，在修复后的评测框架下，重新审视视觉信息的作用位置，证明视觉信号更适合作为生成阶段的门控辅助，而非检索阶段的默认强融合。

在实验结果上，原始 `val-100` 上的最佳历史线 `exp5_densefusion_w045` 仅达到 `Hit@5=0.51`、`Citation@1=0.13`；而在修复后的 `unique-docpage-100 rebuilt` 基准上，文本优先的 `stronger_qia_v6` 检索主线达到 `Hit@5=0.95`、`Precision@5=0.196`、`Citation@1=0.91`，最终端到端主线 `stronger_qia_visualassist_gated_v2` 达到 `Hit@5=0.95`、`Citation@1=0.90`、`EM=0.87`、`ANLS=0.9628`、`Token-F1=0.9518`。本文的主要结论是：在当前课程项目规模与数据条件下，系统性能的上限首先受评测定义与证据建模方式约束，其次才是多模态模块本身；更具体地说，稳定的文本优先检索、严格的 page-level 基准构造以及题型敏感的视觉辅助，比简单叠加更多视觉融合模块更能带来可信且可复现的提升。

## 1. 引言

面向真实文档的问答任务与普通知识库问答存在本质差异。传统文本 RAG 假设知识已被规整为连续文本，检索单元和语义单元高度重合；然而课程讲义、会议材料、作业说明、广告页、扫描件、图表页等文档往往依赖视觉布局传递意义，标题、表头、页码、徽标、图例、局部手写文本和区域位置本身就是答案的一部分。在这样的任务中，如果仍然把系统理解为“先检索一段文本，再让模型回答”，则会遇到三个典型问题。

第一，纯文本抽取会损失版式和局部视觉信息，导致“标题类”“页码类”“图表类”“logo 类”问题无法仅靠语义匹配稳定解决。第二，文档问答需要可追溯性，系统不仅要给出答案，还要能够回到原始文档页、原始 block，甚至原始区域，否则所谓 grounding 只是形式上的。第三，多模态链路的误差来源比普通聊天系统更复杂，数据切分、OCR 质量、块级 schema、索引策略、重排逻辑、评测粒度和生成提示词都有可能改变最后的实验结论。

本项目最初试图回答一个较直接的问题：在一个课程规模的多模态 Doc RAG 系统中，视觉增强能否显著改善检索和问答质量？但随着实验深入，我们逐步意识到一个更值得研究的问题：当系统面对重复页面、通用问题模板和布局敏感问题时，性能瓶颈究竟来自模型能力不足，还是来自 benchmark 本身与真实任务定义不一致？也正因此，本报告的重点不只在于展示最终分数，而更关注系统如何通过一系列问题暴露、假设修正与实验重构，逐步逼近一个可信的结论。

## 2. 研究目标与研究问题

结合仓库中的 `README.md`、`docs/data_indexing.md`、`docs/evaluation_protocol.md`、`docs/retrieval_optimization_plan.md` 与 `skills/mllm-course-project-context/references/project_context.md`，本项目的总体目标可以概括为：构建一套面向多页文档和复杂页面的多模态 RAG 系统，并在公开基准上评估其在检索、回答和引用三方面的能力。

为了让报告具有更清晰的研究叙事，本文将整个实验过程抽象为三个研究问题。

**RQ1：原始 `DocVQA val first-100` 上的检索平台期究竟是模型瓶颈，还是 benchmark 定义问题？**

**RQ2：视觉信息在本项目中最有效的接入位置是检索阶段还是生成阶段？**

**RQ3：在修复 benchmark 后，系统的剩余错误主要来自普通排序问题，还是来自 query grounding 与表示能力缺口？**

围绕这三个问题，项目的思路不是一次性设计一个“完整答案”，而是通过工程实现、误差分析和多轮对比实验逐步缩小解释空间。换言之，本项目的实验价值并不只是“哪条线最好”，而是“为什么旧结论不可信，以及为什么新结论更可信”。

## 3. 系统设计

### 3.1 总体架构

系统整体由四个层次组成。

1. 数据准备与解析层：将公开数据集或原始 PDF/图像文档标准化，并通过 MinerU 产生结构化 block。
2. 分块与索引层：将 block 转换为统一 chunk schema，并构建文本索引与视觉元数据索引。
3. 推理服务层：提供 `/api/v1/chat`、文本 embedding 与视觉 embedding 接口，实现 `query -> retrieve -> generate -> citations` 的主链路。
4. 评测与展示层：运行 retrieval/RAG 两类 benchmark，输出 sample-level 记录、summary 报告以及论文级图表。

对应到仓库实现，这四个层次主要体现在以下文件中：

1. 数据与解析：`scripts/01_prepare_datasets.py`、`scripts/06_run_mineru.py`
2. 分块与索引：`scripts/07_parse_and_chunk.py`、`scripts/08_build_indexes.py`
3. 检索与服务：`src/models/retrieval.py`、`src/serving/api.py`、`src/serving/deps.py`
4. 评测与资产生成：`src/evaluation/benchmark.py`、`scripts/12_run_benchmark_eval.py`、`scripts/16_generate_benchmark_assets.py`

### 3.2 索引对象选择：为什么必须是 block-first 而不是 QA-first

项目在数据建模上的一个关键决定，是明确将 MinerU 输出的 block 视为默认索引对象，而不是把评测集里的 `question` 或 `answers` 人造写入检索库。`docs/data_indexing.md` 和 `docs/项目部署流程.md` 已对此做出明确说明：在真实 RAG 场景中，进入索引的是文档页面经解析后得到的 `text/table/figure/formula/page_image` 等块，而 `question` 对应用户 query，`answers` 只用于评测。

这一选择的研究意义大于工程意义。若将 QA 文本写入索引，检索器实际上是在记忆 benchmark 模板，而不是在匹配文档证据。这样得到的提升很可能只是“任务泄漏式提升”，无法解释真实系统面对新文档时的行为。相反，block-first schema 虽然在早期效果不一定更好，却能保证检索对象与真实部署对象一致，也能保留 `block_id`、`page_no`、`bbox`、`source_path`、`image_path` 等回溯字段，为后续 grounding 提供真正可用的证据链。

### 3.3 Chunk schema 与证据可回溯性

`scripts/07_parse_and_chunk.py` 将标准化样本和 MinerU block 写成统一的 `documents.jsonl` 与 `chunks.jsonl`。在这个过程中，每个 chunk 都带有：

1. `chunk_id`：检索主键
2. `document_id` 与 `block_id`：回到父文档与原始 block
3. `page_no` 与 `bbox`：回到具体页面与区域
4. `source_ref`、`source_path`、`image_path`：支撑引用展示与视觉辅助
5. `metadata`：记录 parser 类型、原始块类型、文档页信息、问题类型等扩展字段

这套 schema 后来被证明至关重要。它不仅服务于“展示 citation”，更服务于误差诊断。例如，项目后期的 `doc-page` 粒度评测、重复页识别、query-image-aware rerank、视觉生成辅助门控，全部依赖这些元数据字段。如果没有统一 schema，后续的所有研究判断只能停留在分数层面，无法解释错误的来源。

### 3.4 API 设计与最小闭环

`src/serving/api.py` 提供了系统的核心 API，其中 `/api/v1/chat` 接收 query、可选的图像数据和上下文，调用 `LocalTextRetriever` 得到证据，再将证据拼接进提示词交由 VLM 回答，并以 `chunk_id/source/page/snippet` 形式返回引用。根据 `docs/B_检索与推理TODO检查清单.md`，这一阶段完成了项目的第一个重要里程碑：从“只有界面和模型封装”走到“有真实 citations 的 text-retrieval 最小闭环”。

这个阶段的贡献看似基础，但实际上它决定了项目后续能否做严肃评测。没有真实的 citations，就无法衡量 `Citation@1`；没有真实 retrieval path，就无法区分检索错误和生成错误；没有配置化的 `top_k_text`、`context_max_chars`、`default_temperature`、`default_max_tokens` 等参数，就无法保证实验可复现。

### 3.5 检索设计：从稀疏匹配到 query-aware rerank

当前检索主线的主体实现在 `src/models/retrieval.py`。系统首先利用稀疏匹配对候选块进行初筛，再通过启发式重排进行更细粒度排序。后期逐步引入的重排信号主要包括：

1. 针对 layout、page number、heading、title、logo 等问题类型的 query-aware rerank
2. 针对 table/chart 问题的结构化数值文本加分
3. 针对页码样式块、短标题块、header/title 块的专门偏置
4. 对低信息块、通用 region 占位符、过长噪声块的惩罚
5. 可选的 duplicate-aware diversification，包括文本指纹惩罚、doc-page 重复惩罚、同 sample 重复惩罚

这些逻辑表面上看较“工程化”，但它们背后有一个明确的研究假设：文档问答中的错误不是均匀分布的，不同问题类型依赖的证据形态不同，因此统一的 lexical similarity 无法稳定解决 layout-sensitive 和 structure-sensitive 问题。后续实验也表明，这一假设是合理的，但其收益建立在 benchmark 定义正确的前提下。

## 4. 实验设置

### 4.1 数据与实验对象

项目的公开实验主要基于 DocVQA，部分脚本也支持 ChartQA。本文聚焦于最终形成完整分析链的 DocVQA 结果。根据 `docs/evaluation_protocol.md` 与 `outputs/eval/docvqa_val_100_benchmark_record.md`，项目经历了两个重要的 benchmark 阶段：

1. 原始历史基准：`DocVQA val first-100`
2. 修复后的主基准：`unique-docpage-100 rebuilt`

其中，第一个基准主要承担“历史基线”和“失败暴露器”的角色；第二个基准才是项目后期用于形成论文结论的主 benchmark。

### 4.2 指标体系

根据 `src/evaluation/benchmark.py` 与 `src/evaluation/metrics.py`，本文采用以下指标：

1. `Hit@5`：Top-5 中是否出现正确证据
2. `Precision@5`：Top-5 中相关证据比例
3. `Citation@1`：首条引用是否正确
4. `Exact Match`：答案与标准答案是否严格一致
5. `ANLS`：近似归一化编辑距离相似度
6. `Token-F1`：答案 token 级重合度

在实现上，项目还显式区分了两种 relevance 判定方式：

1. `sample` 粒度：只认同一 `sample_id`
2. `doc_page` 粒度：认同一原始 `(ucsf_document_id, ucsf_document_page_no)`

这一区分在后文会成为 benchmark 修复的核心。

### 4.3 对比实验线路

项目中有三类代表性线路：

1. 历史纯文本主线：`promptfix_v3`
2. 历史视觉检索融合主线：`exp5_densefusion_w045`
3. 修复基准后的文本优先主线与视觉辅助主线：
   - `stronger_rebuilt`
   - `stronger_qia_v6`
   - `stronger_qia_base`
   - `stronger_qia_visualassist_gated_v2`

这些线路并不是孤立试验，而是围绕研究问题逐步演化的。尤其需要强调的是，后期主线和早期主线并不共享同一个 benchmark 语义，因此不能简单把所有分数线性比较，而必须解释分数背后的评测对象变化。

## 5. 从失败到重构：问题发现与分析过程

### 5.1 原始 `val-100` 的平台期不是一个简单的“效果差”

在历史基准上，最佳 retrieval 结果仅有：

| line | Hit@5 | Precision@5 | Citation@1 |
| --- | ---: | ---: | ---: |
| `promptfix_v3` | 0.49 | 0.114 | 0.12 |
| `exp5_densefusion_w045` | 0.51 | 0.118 | 0.13 |

如果只从结果表看，一个直接结论会是：“系统效果很差，需要更强模型或更强视觉分支。”但项目没有停在这个解释上，而是继续问：为什么 `Hit@5` 只有 0.5 左右？到底是文档证据没被召回，还是命中了相同页面却被评测算错？

### 5.2 误差分析揭示：主要矛盾并不完全在模型

`outputs/eval/docvqa_val_100_error_analysis_exp5.md` 给出的错误分布如下：

1. `true_miss`: 43
2. `ranking_issue`: 38
3. `same_page_false_negative`: 6
4. `generation_issue`: 3
5. `clean_hit`: 10

这一分布具有两个重要含义。

第一，项目的主瓶颈确实在 retrieval，而不是 generation。仅有 3 个 `generation_issue`，说明大多数错误在回答阶段之前就已经决定了。第二，错误并不只是“没检到”，还有相当多是“检到了相同页却不算命中”或“正确 sample 在 top-k 但不是 top-1”。这意味着仅靠继续提升语言模型生成能力并不能解决主问题。

更关键的是，这份分析首次显式引入了 `same_page_false_negative` 标签：如果 top-k 命中了同一原始文档页，但没有命中当前 sample，本质上这是 benchmark 的粒度冲突，而不完全是模型失效。

### 5.3 重复页污染：benchmark 的隐性病灶

根据 `docs/unique_docpage_100_protocol.md` 和 `outputs/eval/final_assets/README.md`，原始 `DocVQA val first-100` 中共有 100 个 QA 样本，但只对应 29 个唯一文档页，存在 71 个重复页问题。这一事实彻底改变了我们对原始 benchmark 的理解。

在这种数据分布下，检索系统会遭遇一种结构性不公平：对于诸如 “What is the title of the document?”、“What is the page no mentioned in this document?”、“Which college's name is specified in the logo?” 这类问题，多个样本的 query 本身高度相似，甚至直接相同；而不同 sample 又共享同一原始页面的视觉布局和局部文本。于是，系统即便命中了一张正确页面，也可能由于 `sample_id` 不一致而被算错；反过来，系统若利用了 benchmark 模板中的高频词，也可能获得虚高的 lexical overlap。

这说明原始 `val-100` 不是一个纯粹测量“page-level retrieval”能力的基准，而是把 page retrieval、sample disambiguation 和 duplicate collision 混合在了一起。继续在这个 benchmark 上调参，容易把评测偏差误认为检索能力问题。

### 5.4 为什么“修 benchmark”比“调模型”更重要

面对这一问题，项目没有选择继续堆叠更强的视觉融合或更复杂的启发式，而是优先修正 benchmark。本质上，这是一种研究判断：若评测对象本身不能忠实反映任务目标，再复杂的模型优化也难以得出稳定结论。

于是，项目新增 `scripts/15_build_unique_docpage_benchmark.py`，从 `data/processed/docvqa/val.jsonl` 中按 `ucsf_document_id + ucsf_document_page_no` 分组，保留每页一个代表样本，并按问题类型分层抽样，最终构造出 100 个唯一页面组成的固定 manifest。更重要的是，项目没有只替换评测样本，而是进一步重建了与该 manifest 对齐的 documents、chunks 和 indexes。也就是说，修复不仅发生在 evaluator，也发生在 retrieval corpus 本身。

这一步具有决定性意义。若只替换测试集而保留旧语料，仍然可能被质疑“评测对象变了但索引对象没变”；而重建后的 `unique-docpage-100 rebuilt pipeline` 则更接近真正的“同分布训练-检索-评测”闭环。

## 6. 修复基准后的系统再评估

### 6.1 检索结果的跃迁及其因果解释

在修复后的 benchmark 上，主要 retrieval 结果如下：

| line | Hit@5 | Precision@5 | Citation@1 |
| --- | ---: | ---: | ---: |
| `stronger_rebuilt` | 0.93 | 0.19 | 0.89 |
| `stronger_qia_v6` | 0.95 | 0.196 | 0.91 |
| `exp5_rebuilt` | 0.93 | 0.19 | 0.88 |

这一跃迁不能简单解释为“新模型更强”。更合理的解释是三个因素共同作用：

1. **评测对象与任务对象重新对齐。**
   原先一个页面对应多个 sample，如今一页只对应一个代表问题，sample-level 评价不再系统性惩罚正确的 page retrieval。

2. **索引语料与固定 manifest 对齐。**
   重建后的 corpus 避免了旧全量切片中的重复页混淆，使 chunk 排序问题更接近真实 page ranking，而不是 sample collision。

3. **重排逻辑终于开始服务真实错误类型。**
   在 benchmark 正确之后，`layout`、`title`、`page number`、`table/list` 等题型上的 query-aware rerank 才真正体现出价值，而不是被重复页噪声淹没。

这里最值得强调的是：修复 benchmark 后，文本优先的 `stronger_qia_v6` 反而优于历史上看起来更有前景的 visual dense fusion 线。这表明早期 `exp5` 的收益中，至少有一部分来自旧 benchmark 的偶然适配，而不是可泛化的架构优势。

### 6.2 进一步诊断：剩余错误的性质已经改变

`outputs/eval/docvqa_val_unique_docpage_100_top1_error_analysis_stronger_v2.md` 进一步揭示了一个很有价值的现象：在 repaired benchmark 上，最初的 top-1 错误并不主要来自真实 OCR 排序失灵，而是来自 synthetic `fallback_text` chunk 对 lexical overlap 的劫持。换言之，检索器有时并不是“没读懂页面”，而是被 `Question: ...` 风格的回退文本抢走了 top-1。

为此，项目在 `src/models/retrieval.py` 中对 `rerank_profile == "stronger"` 的稀疏预排序做了一个看似小但实际非常关键的改动：在计算 query 稀疏相似度前，先去掉低信息 stopwords。这一改动的本质，不是“多了一个预处理技巧”，而是将检索关注点从“整句问题模板重合”重新拉回“页面内容词重合”。

改动后，`stronger_rebuilt` 进一步提升为 `stronger_rebuilt_top1v2`，其结果达到：

1. `Hit@5`: `0.93 -> 0.95`
2. `Precision@5`: `0.19 -> 0.196`
3. `Citation@1`: `0.89 -> 0.90`

这个结果非常关键，因为它说明修复 benchmark 后，错误结构已经从“评测偏差主导”转移为“query grounding 与表示差距主导”。也就是说，项目终于进入了一个更真实、更值得继续优化的状态：系统剩余的难题开始更像真正的文档问答难题，而不是 benchmark 假象。

## 7. 视觉信息应该放在哪里：检索融合 vs 生成辅助

### 7.1 为什么 retrieval-time visual fusion 没有成为最终主线

在原始 benchmark 上，`exp5_densefusion_w045` 的确带来了从 `Citation@1=0.12` 到 `0.13` 的小幅提升，因此它曾被视为一个 promising direction。然而，修复 benchmark 后，对应的 `exp5_rebuilt` 却没有超过文本优先主线，反而在 top-1 citation 上落后于 `stronger_qia_v6`。

这一结果说明，视觉描述向量虽然能提供“相似页面”的弱信号，但在当前数据规模下，它尚不足以稳定区分语义相近、布局相似却来源错误的页面。特别是在广告页、表格页、公司材料等模板化页面中，视觉相似性反而可能将检索引向错误页面。

换句话说，retrieval-time visual fusion 在本项目中的主要问题不是“完全无用”，而是“噪声成本高于稳定收益”。当任务目标是首条引用正确、证据可追溯时，这种不稳定性比小幅 recall 增益更值得警惕。

### 7.2 生成阶段视觉辅助的理论动机

如果视觉信息不适合主导检索，那么它还剩下什么价值？项目后期给出的答案是：视觉信号更适合在检索已经给出稳定候选之后，用于补充局部页面判别和短文本识别。

这一路线的理论动机很清楚。对于诸如标题、logo、页码、包装文字、局部手写体等问题，真正困难的部分不是“去哪一页找”，而是“这页的哪个局部才是答案”。文本优先检索已经能把正确页面召回，视觉信号此时更像一个局部识别器，而不是一个全局排序器。

基于这一理解，`src/serving/api.py` 中引入了 `_build_generation_visual_assist` 机制：当 `generation_visual_assist` 启用且 `_should_enable_visual_assist` 判定题型适合时，系统将为生成模型附加裁剪图、页面图和视觉 descriptor 文本，但不改变引用排序。

### 7.3 为什么必须门控

项目很快发现，视觉辅助不是“加上就更好”。在 `outputs/eval/docvqa_val_unique_docpage_100_visual_assist_experiment.md` 中，第一版无差别视觉辅助仅让 `Exact Match` 从 `0.85` 升到 `0.86`，却让 `ANLS` 与 `Token-F1` 下降。说明额外视觉信息会改变模型关注点，一旦题型并不依赖视觉局部辨识，图像上下文反而会制造噪声。

因此，项目进一步引入门控策略，仅对以下题型启用视觉辅助：

1. `logo`
2. `heading`
3. `title`
4. `page number`
5. `pack text`
6. 短手写文本

同时，对 `chart`、`table`、`how many`、`value`、`abstract` 等题型禁用视觉辅助。这一门控不是经验主义的小修小补，而是一个研究结论：视觉证据的有效性高度依赖题型，如果不显式建模这一条件，多模态输入会从“补充信息”退化为“额外噪声”。

### 7.4 最终结果：视觉信息更适合作为 generation-time gated assistance

采用门控后的最终端到端结果如下：

| line | Hit@5 | Citation@1 | EM | ANLS | Token-F1 |
| --- | ---: | ---: | ---: | ---: | ---: |
| `stronger_qia_base` | 0.95 | 0.90 | 0.85 | 0.949966 | 0.936036 |
| `stronger_qia_visualassist_gated_v2` | 0.95 | 0.90 | 0.87 | 0.962800 | 0.951780 |

这个结果的含义比“分数提升了 0.02”更丰富。首先，检索 grounding 指标不变，说明视觉辅助没有破坏 citation stability。其次，`EM`、`ANLS`、`Token-F1` 同时改善，说明视觉提示确实帮助模型在需要局部页面识别的题型上生成更贴近 gold answer 的输出。最后，这一提升是在“不改变主证据排序”的前提下获得的，因此它更容易解释，也更适合被保留为最终系统设计。

据此，本文对 **RQ2** 的回答是：在当前项目条件下，视觉信息最有效的使用位置不是 retrieval-time dense fusion，而是 generation-time gated assistance。

## 8. 结果汇总与核心结论

### 8.1 历史 benchmark 与修复 benchmark 的关系

根据 `outputs/eval/docvqa_val_100_benchmark_record.md`，项目后期已明确区分两类 benchmark 的职责：

1. 原始 `val-100`：历史基线与失败暴露器
2. `unique-docpage-100 rebuilt`：后期主 benchmark 与正式结论依据

这一区分本身就是论文叙事的重要组成部分。它避免了一个常见问题：把“旧 benchmark 上的最好分数”误写成“整个项目的最终结论”。实际上，项目最终最有价值的工作，恰恰是证明旧 benchmark 的解释力不足，并用新的 benchmark 替代它。

### 8.2 对三个研究问题的回答

**RQ1：原始 `val-100` 的平台期是模型瓶颈还是 benchmark 问题？**

答案是两者都有，但 benchmark 有效性问题是主导因素。重复页污染使 sample-level 指标严重低估了 page-level retrieval 能力，从而让大量看似“模型失败”的案例实际上带有评测偏差。

**RQ2：视觉信息应放在检索阶段还是生成阶段？**

在当前项目条件下，视觉信号作为 retrieval-time dense fusion 的收益不稳定；而作为 generation-time gated assistance 时，则能在不伤害 grounding 的前提下稳定提升答案质量。因此，生成阶段门控视觉辅助是更优选择。

**RQ3：修复 benchmark 后，剩余错误来自哪里？**

剩余错误不再主要来自普通排序问题，而更多来自两类更真实的难点：一类是 generic/deictic query 缺乏 query-side grounding，另一类是 OCR/实体识别/布局表示本身仍存在能力缺口。这意味着系统已经从“benchmark 噪声主导阶段”进入“真实文档理解主导阶段”。

### 8.3 本项目的真实贡献

如果仅从实现角度总结，本项目的贡献可能只是“做了一个多模态 RAG demo”。但从研究过程角度看，其更真实的贡献包括：

1. 构建了 block-first、citation-aware、可复现的多模态文档问答工程闭环
2. 通过误差分析识别出旧 benchmark 的重复页病灶，并提出固定 manifest + rebuilt corpus 的修复方案
3. 证明视觉信息的最优接入方式依赖任务位置，不应默认在检索阶段强融合
4. 将项目的最终结论建立在“评测修复后仍然成立”的实验线上，而不是建立在偶然适配旧 benchmark 的历史高点上

## 9. 效度威胁与局限性

一份科研规范的报告不能只写“我们做到了什么”，也必须明确“哪些结论暂时不能过度外推”。本项目至少存在以下四类效度威胁。

### 9.1 外部效度

当前主要结论建立在 DocVQA 风格数据上，尤其集中于 `unique-docpage-100` 固定切片。尽管项目脚本也支持 ChartQA 和普通 PDF，但“视觉信息更适合 generation-time gating”这一结论能否无缝推广到更多开放域、多页长文档或手写密集页面，仍需更多实验支撑。

### 9.2 内部效度

项目中的若干 retrieval gain 来自启发式重排逻辑，这些规则对当前 benchmark 很有效，但仍可能与当前数据分布高度耦合。因此，虽然它们能支撑课程项目结论，却未必代表一种对任意数据都最优的普适检索器。

### 9.3 构念效度

尽管 `unique-docpage-100` 已显著优于原始 `val-100`，但它依然主要衡量的是 page-level retrieval，而不是更细粒度的 region-level grounding。对于某些答案只对应页面局部区域的任务，page-level 正确并不等于 region-level 正确。

### 9.4 工程效度

当前主线更偏向“离线索引 + 在线问答 + 标准评测”的科研样机，而不是完整产品。用户上传文件后的即时解析、增量建索引和在线文档管理虽然已有 workspace 方向的代码基础，但仍未构成本文实验的主评估对象。因此，本文结论更多反映系统架构与评测设计的研究价值，而不是完整产品能力。

## 10. 后续工作

基于现有发现，后续工作可以沿三条线继续展开。

第一，推进在线文档接入与增量索引，使当前“离线 benchmark 驱动”的系统进一步走向真实可交互应用。第二，将当前 query-aware rerank 中的启发式规则进一步学习化，例如引入 page-level reranker 或 region proposal 机制，使系统对 generic title/page-number 问题有更强 query grounding 能力。第三，在视觉辅助上继续细化门控策略，例如结合置信度估计，只在模型对局部文字读取明显不确定时触发图像补充，以进一步控制多模态噪声。

## 11. 结论

本项目最终形成的最重要认识，并不是“多模态比纯文本更强”这样一个宽泛命题，而是一个更具体也更有条件约束的结论：在课程规模的文档问答系统中，**首先要保证评测对象、索引对象与任务对象的一致性，其次才谈多模态增强是否有效**。原始 `val-100` 上看似持久的性能平台期，经过误差分析后被证明很大程度上是 benchmark 病灶；修复 benchmark 并重建语料后，文本优先的检索主线获得了显著提升，而视觉信息在生成阶段、以门控方式介入时才真正展现出稳定价值。

因此，本文认为该项目的核心贡献不应被概括为“实现了一个多模态 RAG 原型”，而应被概括为：**通过一次从工程闭环到评测重构再到架构重定位的实验过程，澄清了多模态文档问答系统中什么才是主要矛盾，以及什么样的改进才值得被视为可信的改进。**

## 相关工作

多模态文档问答与传统文本 RAG 的关键差别，在于检索对象不再只是线性文本，而是同时包含页面版面、表格区域、图形区域、页码、标题与局部视觉证据。现有路线大体可以分为三类。第一类是 OCR-first 文档问答系统，即先把页面转写成文本，再执行检索与生成；这类方法工程实现较直接，但容易损失布局与局部视觉信息。第二类是端到端视觉语言模型路线，直接把整页图像送入模型完成问答；其优势是视觉保真更强，但引用稳定性、可解释性和实验复现成本往往更高。第三类是 block-first 多模态 RAG 路线，即先进行页面解析，再把文本块、表格块、图形块、页面图像块统一纳入检索与 grounding 框架，本文工作属于这一类。

与传统纯文本 RAG 相比，本文方法并不把“回答正确”作为唯一目标，而是同时要求系统保留 `page_no`、`bbox`、`source_ref`、`image_path` 等回溯字段，使输出可以回到具体页面与证据块。与默认在检索阶段强融合视觉向量的方法相比，本文实验更支持“文本优先检索 + 生成阶段视觉辅助”的策略：在当前课程项目的数据规模与误差结构下，稳健的文本检索和基准修复比简单堆叠视觉融合更能带来可信提升。这一点与仓库内的 benchmark 修复、误差分析与 visual assist 对比实验保持一致。
