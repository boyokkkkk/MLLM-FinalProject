# 多模态文档理解与检索问答系统结题报告

陈宝怡 23336029 孙韵淇 23336215 吴承徽 23336247

## 摘要

本项目围绕教育学习、学术研究及日常办公场景中基于复杂文档信息的多模态问答任务，设计并实现了一套以结构化解析、块级索引、文本优先检索、视觉辅助生成与可追溯引用为核心的多模态检索增强生成系统。项目的初始目标，是验证“文档结构感知 + 多模态检索增强”能否有效缓解传统纯文本问答系统在复杂页面理解、长文档检索与证据定位方面的局限；而在具体实现与实验推进过程中，我们进一步发现，真正决定系统结论是否可信的关键，不仅在于模型或模块是否足够复杂，更在于评测对象、索引对象与任务定义三者是否严格对齐。

围绕这一认识，本文完成了三方面工作。其一，构建了从数据标准化、MinerU 页面解析、块级语义切分、轻量索引构建、后端 `/api/v1/chat` 问答服务到 benchmark 评测的完整工程闭环。其二，在对原始 `DocVQA val first-100` 子集进行误差分析时，识别出“100 个样本仅对应 29 个唯一文档页”的重复页污染问题，并据此重新构建 `unique-docpage-100` 固定评测清单，以及与该清单严格对齐的 rebuilt corpus。其三，在修复评测协议后重新审视视觉信息在系统中的作用位置，证明视觉信号在当前项目条件下更适合作为生成阶段的门控辅助，而不宜默认前置为检索阶段的强融合信号。

实验结果表明：在原始 `val-100` 上，历史最佳检索线 `exp5_densefusion_w045` 仅取得 `Hit@5=0.51`、`Citation@1=0.13`；而在修复后的 `unique-docpage-100 rebuilt` 基准上，文本优先主线 `stronger_qia_v6` 达到 `Hit@5=0.95`、`Precision@5=0.196`、`Citation@1=0.91`，最终端到端主线 `stronger_qia_visualassist_gated_v2` 达到 `Hit@5=0.95`、`Citation@1=0.90`、`EM=0.87`、`ANLS=0.962800`、`Token-F1=0.951780`。研究结果说明，在课程规模的多模态文档问答系统中，稳定的文本优先检索、合理的页级评测构造以及题型敏感的视觉辅助机制，比单纯堆叠更多多模态融合模块，更能带来可信、可解释且可复现的系统提升。

## 1 引言

### 1.1 研究背景

在教学、科研与办公场景中，PDF 讲义、PPT 截图、扫描页、图表页、表单页等文档广泛存在。此类文档往往同时包含文本、表格、图像、公式、标题层级与版面布局等多种信息，其语义不仅来自文字内容本身，也来自页面结构、局部视觉线索与阅读顺序。用户在复习、问答、检索和跨页理解时，通常希望系统能够快速定位与问题对应的证据页面、证据区域与关键短语，而非仅返回模糊的“相关文本”。

传统基于纯文本切分的问答系统在此场景中存在天然不足。首先，纯文本抽取会丢失图表、标题、徽标、页码、表格边界等非线性语义信号，使得页面在被“线性化”后发生信息坍缩。其次，长文档往往超出单次上下文窗口，若检索机制缺乏对结构与题型的感知，生成模型即便具备较强语言能力，也难以稳定地回到真正相关的页面与区域。再次，文档问答不仅要求“答对”，还要求“答得可追溯”，即系统应尽量返回可验证的 citation，而非不可解释的自由生成结果。

随着多模态大模型与检索增强生成技术的发展，文档问答逐步从“整页 OCR 后交给语言模型”转向“结构化解析 + 块级检索 + 引用式生成”的体系化路线。然而，现有方法在实际工程落地时仍面临两个关键问题：一是视觉信息究竟应主要作用于检索阶段还是生成阶段；二是实验结论是否建立在与任务目标一致的评测协议之上。本文的结题工作，正是在这一背景下展开。

### 1.2 研究目标与核心任务

本研究旨在构建一个面向课程文档的多模态智能问答系统，实现对复杂页面的结构化解析、块级证据检索、可追溯回答生成与标准化实验评测。结合开题阶段的设想与后续实现演化，本文的核心任务可概括为以下四项：

1. 构建文档结构化解析与块级语义表示流程。围绕 `scripts/01_prepare_datasets.py`、`scripts/06_run_mineru.py` 与 `scripts/07_parse_and_chunk.py`，将原始 DocVQA/ChartQA 数据与用户文档统一转换为块级语义对象。
2. 建立统一的索引与检索主线。围绕 `scripts/08_build_indexes.py` 与 `src/models/retrieval.py`，实现以文本优先为主、视觉元数据为辅的证据召回与重排机制。
3. 构建从检索到回答再到引用返回的服务闭环。围绕 `src/serving/api.py`，打通 `/api/v1/chat` 的 `query -> retrieve -> generate -> citation` 主流程。
4. 建立可复现的 benchmark 评测与误差分析框架。围绕 `scripts/12_run_benchmark_eval.py`、`scripts/14_analyze_benchmark_errors.py`、`scripts/15_build_unique_docpage_benchmark.py` 与 `src/evaluation/benchmark.py`，形成实验记录、指标输出与问题诊断闭环。

### 1.3 研究问题

在系统实现与实验推进过程中，本文将项目目标进一步凝练为三个研究问题：

1. 原始 `DocVQA val first-100` 上的长期平台期，主要源自模型与检索能力不足，还是源自 benchmark 的任务定义偏差。
2. 在当前课程项目的数据规模与系统架构下，视觉信息最有效的接入位置究竟是检索阶段，还是生成阶段。
3. 在修复 benchmark 之后，系统剩余错误的主要来源，究竟是普通排序噪声，还是更深层的 query grounding、不充分 OCR 表示与版面理解缺口。

相比仅回答“哪条实验线得分最高”，本文更关心“哪些提升是真提升、哪些提升只是对旧 benchmark 的偶然适配，以及哪些失败属于模型问题、哪些失败属于评测定义问题”。这也是本结题报告相较开题报告更重要的认识增量所在。

### 1.4 研究推进主线

为了避免将本文写成简单的“功能清单”或“实验流水账”，有必要先明确本研究的推进逻辑。整体而言，本文的研究过程并非沿着一条线性“实现系统并持续提分”的路径展开，而是经历了一个不断修正问题定义的过程：项目初期，我们将主要矛盾理解为“纯文本方法缺少视觉信息，因此需要更强的多模态融合”；中期，当系统主链路和初步 benchmark 建立后，我们观察到性能提升并不稳定，且不同实验线之间的差异难以形成可信解释；后期，通过误差分析与 benchmark 审视，我们意识到问题的核心并不首先是“模态不够多”，而是“评测对象、索引对象与证据定义并未充分对齐”。

基于这一认识，本文的叙事主线被重新组织为五个阶段：第一，完成从解析、检索到回答与 citation 返回的最小工程闭环；第二，在旧 benchmark 上识别异常现象而非急于接受表面分数；第三，通过重复页污染分析重新界定主要研究问题；第四，在修复 benchmark 后重新比较文本优先路线与视觉路线的真实收益；第五，将视觉信息从检索期强融合重定位为生成期门控辅助。换言之，本文真正要回答的不是“我们最终用了哪些模块”，而是“为什么系统会从最初设想演化到现在的主线形态，以及这种演化如何被实验事实所支持”。

## 2 相关工作

### 2.1 多模态大模型与文档理解

多模态大模型为文档理解提供了新的技术基础。与只处理线性文本的语言模型相比，视觉语言模型能够同时处理页面图像与文字内容，因此在表单识别、图文推理、图表问答和页面短文本辨识等任务上具有明显优势。对于文档场景而言，这种优势尤其体现在标题、徽标、局部短语、页码与图表区域的识别上。

然而，多模态能力并不自动等价于可用的文档检索系统。若直接将整页乃至整本文档输入视觉语言模型，一方面会受到上下文长度与推理成本约束，另一方面也无法自然地解决“先找哪一页、再读哪一块”的证据组织问题。因此，多模态模型更适合作为文档问答链条中的推理器或局部视觉补偿器，而非唯一的信息组织机制。

### 2.2 文档 RAG 与多模态 RAG

RAG 通过“外部检索 + 条件生成”缓解了纯生成模型在长文档和外部知识利用上的限制，已经成为文档问答的重要技术路线。传统文本 RAG 的优点在于检索稳定、成本可控、可解释性较强，但其索引对象通常是线性文本片段，难以显式保留页面结构、图表证据与局部视觉信息。

近年来，多模态 RAG 开始将视觉路径引入检索与生成。VisDoM 强调视觉丰富文档中的多源召回与后融合，VisRAG 强调基于页面图像的检索价值，M3DocRAG 则进一步关注多页、多文档场景下的多模态证据组织。这些工作启发本文认识到：文档问答系统不应简单复制纯文本 RAG，也不应完全依赖整页视觉推理，而应在检索粒度、语义保持与可追溯性之间寻找工程上更稳健的平衡点。

### 2.3 结构感知切分与块级证据表示

除模型层面外，文档问答中的另一个关键问题在于切分。若切分方式不能保留版面结构与阅读顺序，即便检索命中了“相关文本”，也未必命中了“完整证据”。LayoutLMv3 等工作表明，文档的二维结构信息对语义理解具有重要作用；MinerU 等解析工具则从工程实践层面为标题、表格、图像与区域块的统一表示提供了可操作基础。

本文在实现中采用“块优先”思路，将 MinerU 输出视为主索引对象，并在 `chunks.jsonl` 中保留 `chunk_id`、`block_id`、`page_no`、`bbox`、`source_ref`、`source_path`、`image_path` 等回溯字段。这样的设计并不只是“切得更细”，而是为后续检索、重排、引用返回与误差分析提供了统一证据载体。

### 2.4 本文定位

综合上述工作可以看到，现有研究往往默认“多模态信息越早介入越好”，但在高度模板化、页面相似度高、重复视觉元素较多的课程文档场景中，这一前提未必成立。此外，若 benchmark 的构造方式与真实任务目标不一致，则实验结论也可能发生系统性偏移。

因此，本文的定位并不是提出一种全新的基础模型，而是在课程项目约束下构建一条更可诊断、可验证、可复现实验结论的系统路线：以块优先结构为基础，以文本优先检索为主路径，以门控视觉辅助为补充路径，并通过 benchmark 修复保证结论的有效性。

## 3 系统设计与实现

### 3.1 总体设计思路

针对开题报告中提出的“语义丢失”与“长文档检索困难”两类核心问题，本文最终采用“结构化解析 + 块级表示 + 文本优先检索 + 视觉辅助生成 + 标准化评测”的总体路线。与开题阶段偏重“双路径 RAG + 晚融合”的设想相比，结题阶段最大的变化不在于模块数量的增减，而在于系统内部主次关系的重新排序：文本检索被确立为主证据排序机制，视觉信息则更多承担局部消歧与回答补偿的角色。

系统整体可划分为四个阶段：

1. 数据标准化与页面解析。
2. 块级语义对象构建与索引生成。
3. 证据召回、重排与组织。
4. 回答生成、citation 返回与离线评测。

这一设计的核心原则是：任何生成结果都应尽量能够回到具体页面与具体块，任何实验结论都应尽量能够回到明确的索引对象与评测协议。

更进一步地看，这一总体设计并非一开始就完全确定，而是在实验过程中被不断校正。也就是说，第 3 节所呈现的系统架构，不应被理解为先验设定好的唯一方案，而应被理解为在多轮失败分析之后沉淀下来的“当前最合理结构”。因此，系统设计本身也是研究结论的一部分：它记录了项目如何从“追求更多模态融合”转向“优先保证证据定义、检索稳定性与引用可验证性”。

### 3.2 数据标准化、MinerU 解析与块优先结构

数据入口由 `scripts/01_prepare_datasets.py`、`scripts/06_run_mineru.py` 与 `scripts/07_parse_and_chunk.py` 共同构成。前者负责将 DocVQA、ChartQA 等评测集统一标准化为 JSONL；中者负责将页面图片或 PDF 输入交给 MinerU，得到归一化的页面块；后者负责将解析结果转换为 `documents.jsonl` 与 `chunks.jsonl` 两类产物。

其中，`documents.jsonl` 主要记录文档级与样本级元信息，不直接作为检索对象；真正进入索引的是 `chunks.jsonl` 中的块级对象。每个块至少保留以下字段：`chunk_id`、`document_id`、`block_id`、`page_no`、`bbox`、`source_ref`、`source_path`、`image_path` 与 `metadata`。项目在后期已明确放弃将 benchmark 问题文本作为默认检索对象的做法，仅将其保留为早期 smoke test 所需的可选调试机制。这一转变意味着系统从“记住问题”走向“读取页面”，从而保证检索对象与真实任务对象保持一致。

值得强调的是，`scripts/07_parse_and_chunk.py` 在后续优化中还继续传播了 `ucsf_document_id`、`ucsf_document_page_no` 与 `question_types` 等字段，使得文档页级分组、重复页抑制、页级重排和诊断分析具备了实现基础。这说明结构化 schema 的价值不仅在于支持运行，更在于为后续研究问题的细化提供数据接口。

### 3.3 索引构建与文本优先检索主线

`scripts/08_build_indexes.py` 负责基于块级对象构建本地轻量索引。当前主线索引主要包含三类对象：一是以 `doc_store.json`、`postings.json` 与 `document_frequency.json` 为核心的稀疏文本索引；二是 `visual_store.json` 形式的视觉元数据索引；三是部分实验中使用的视觉描述符存储与向量文件。

核心检索逻辑位于 `src/models/retrieval.py`。系统先通过稀疏相似度进行候选筛选，再根据 `rerank_profile` 进入更强重排。后期逐步加入的重排特征包括：针对 `title`、`heading`、`logo`、`page number` 等题型的 query-type-aware rerank，针对表格与图表问题的结构化文本偏置，针对短标题块与页码样式块的奖励，以及针对通用噪声块与低信息块的惩罚。其关键思想是：系统不再把“相关性”仅理解为词面重叠，而是显式区分不同题型对证据形态的不同依赖。

特别地，在 `stronger` profile 中，系统在稀疏预排序阶段会先过滤 stopwords，再计算稀疏相似度。该改动针对的是后文将讨论的 `fallback_text` 劫持问题，即某些 synthetic chunk 仅因重复题干文本而在稀疏检索中获得不合理优势。由此可见，检索优化的价值并不只是“加规则”，更在于不断逼近“何为真实证据”的定义。

### 3.4 服务接口、证据组织与引用返回

`src/serving/api.py` 提供完整的后端服务闭环。系统通过 `/api/v1/chat` 接收 query、上下文与可选图片输入，调用 `LocalTextRetriever` 召回证据块，再将证据组织为生成提示词，最后由视觉语言模型生成回答并返回 citation。citation 至少包含 `chunk_id`、`source`、`page`、`snippet` 等字段，并通过 `source_ref` 保留到原始页面与区域的回溯路径。

从实现层面看，后端闭环的意义不只是“系统能跑起来”，而是完成了从可演示 demo 到可评测研究原型的转变。只有当 retrieval、generation 与 citation 返回被统一放入同一主流程后，benchmark 才真正具备比较意义，误差分析也才具备落点。

### 3.5 从检索期视觉融合到生成期门控视觉辅助

项目关于视觉路径的实现经历了明显演化。早期思路倾向于在检索阶段直接进行视觉稠密融合，希望利用页面图像的全局相似性提升召回质量；相应地，`src/evaluation/benchmark.py` 中也实现了视觉融合与 query-image-aware rerank 等实验路径。然而，后续实验表明，在页面布局相近、标题模板重复、通用视觉元素较多的课程文档场景中，视觉描述符容易把“看起来相似”但证据并不正确的页面推到更前位置。

基于这一观察，项目后期将视觉信息更多转移到生成阶段使用。`src/serving/api.py` 中的 `_should_enable_visual_assist` 与 `_build_generation_visual_assist` 共同实现了门控视觉辅助策略：仅在 `logo`、`pack text`、`heading`、`title`、`page number` 与短手写文本等题型上启用视觉辅助；对于 `chart`、`table`、一般数字题与抽象语义题则禁用该机制。该策略保留了视觉信息对局部短文本辨识的价值，同时避免其在不适合的题型上扰动文本证据主线。

## 4 实验设计

### 4.1 数据集与评测对象

本文主要围绕 DocVQA 进行系统性实验，并辅以 ChartQA 的流程验证。项目在 `docs/evaluation_protocol.md` 中明确提出：评测对象不是单纯的“能否对话”，而是完整多模态 RAG 系统在检索、回答与引用三个层面的综合能力。

其中，原始历史基准使用 `DocVQA val first-100`；后期主基准使用 `unique-docpage-100`。后者通过 `scripts/15_build_unique_docpage_benchmark.py` 从 `data/processed/docvqa/val.jsonl` 出发，按 `(ucsf_document_id, ucsf_document_page_no)` 分组，保留每个文档页一个代表性问题，并结合 question type 做分层抽样，最终得到固定的 100 页 manifest。重要的是，项目不仅替换了 eval slice，也同步重建了 `documents`、`chunks` 与 `indexes`，从而形成与 manifest 严格对齐的 rebuilt corpus。

### 4.2 指标体系

本文采用的指标体系包括三类：

1. 检索指标：`Hit@5`、`Precision@5`。
2. 引用指标：`Citation@1`。
3. 回答指标：`Exact Match`、`ANLS`、`Token-F1`，以及部分分析中使用的 `Answer-Contains`。

此外，`src/evaluation/benchmark.py` 中同时保留了 sample-level 与 doc-page-level 两种相关性判定逻辑。前者强调是否命中完全一致的样本来源，后者强调是否命中同一原始文档页。后续实验表明，这一区分不是技术细节，而是理解原始 benchmark 失真的关键前提。

### 4.3 对比实验设置

为分析系统演化中的真实收益，本文重点比较以下代表性实验线：

1. `promptfix_v3`：旧 benchmark 上较稳定的文本基线。
2. `exp5_densefusion_w045`：旧 benchmark 上表现最好的视觉稠密融合线。
3. `stronger_rebuilt`：修复 benchmark 后的文本优先重排主线。
4. `stronger_qia_v6`：加入 query-image-aware rerank 后的当前最佳检索线。
5. `stronger_qia_base`：修复 benchmark 后的 clean grounding-first RAG 基线。
6. `stronger_qia_visualassist_gated_v2`：当前最佳端到端主线。

这些实验线并不只是横向调参结果，而是围绕研究问题逐步缩小解释空间的证据链。

从科研叙事的角度看，这组对比实验实际上对应三轮连续追问。第一轮追问是：在旧 benchmark 上，视觉融合是否真的优于文本优先基线。第二轮追问是：若旧 benchmark 本身存在偏差，那么在修复后的页级基准上，哪条路线还能保持优势。第三轮追问是：当文本证据主线已经较稳定时，视觉信息是否仍能以另一种方式提升最终回答质量。正是这种层层收缩的问题设计，使本文能够从“现象记录”推进到“机制解释”。

## 5 实验过程中的关键问题与解决思路

### 5.1 从 API 骨架到真实 citation 闭环

项目早期最大的工程问题并非模型效果，而是主链路不完整。尽管仓库已有 API、前端与模型封装，但若 retrieval、generation 与 citation 未被统一放入同一条主流程，任何 benchmark 都只能停留在 demo 层面，无法形成可验证的研究结论。

为此，项目首先围绕 `/api/v1/chat` 建立最小可用闭环，并在相关 TODO 文档中固化关键约束，包括：打通 `query -> retrieve -> generate` 主链路、确保 citation 返回真实字段而非占位数据、统一检索与生成配置、在检索失败时提供可控 fallback 行为。该阶段虽然没有直接带来分数提升，却为后续所有科学分析提供了前提。

这一阶段的意义在于，它把“系统是否可研究”与“系统是否可运行”区分开来。一个只能展示回答、却无法说明回答来自何处的系统，很难支撑后续关于 retrieval、grounding 与视觉作用位置的严肃分析。因此，工程闭环不是论文叙事之外的准备工作，而是研究得以成立的第一步。

### 5.2 原始 benchmark 的重复页污染问题

在对 `outputs/eval/docvqa_val_100_error_analysis_exp5.md` 进行误差分析后，项目发现原始 `val-100` 中虽然包含 100 个 QA 样本，但实际上只对应 29 个唯一文档页。这意味着许多不同 sample 共享同一原始页面，而 sample-level 指标会把“命中同页但命中到其他 sample 副本”的情况记为错误。

这一发现实质性改变了项目的研究重点。继续在旧 benchmark 上堆叠视觉模块，已无法回答“系统究竟没有找到正确页面，还是只没有找到当前 sample 的那一份副本”。因此，项目的主要矛盾从“如何继续增加复杂模块”转向“如何首先修复 benchmark 的任务定义偏差”。

### 5.3 固定 manifest 与 rebuilt corpus 的设计

针对重复页污染问题，项目新增 `scripts/15_build_unique_docpage_benchmark.py`，以 `(ucsf_document_id, ucsf_document_page_no)` 作为分组键，仅保留每页一个代表性问题，并通过题型分层采样构造 `unique-docpage-100` 固定评测清单。随后又进一步重建与该清单对齐的 `documents`、`chunks` 与 `indexes`，避免出现“评测切片变了，但检索语料仍保留旧分布”的结构性偏差。

这一设计在方法论上具有重要意义。它表明 benchmark 修复不是“换一个测试集”这么简单，而是必须同步修复评测对象、索引对象与任务定义之间的一致性关系。只有在此基础上，后续分数才真正具备比较价值。

从叙事逻辑上看，这一步构成了整篇论文的转折点。在此之前，项目面对的是“某些实验线为什么提不上去”的局部优化问题；在此之后，项目面对的则是“我们究竟在优化一个什么任务”的更根本问题。正因为完成了这一层问题重定义，后续关于文本主线、视觉辅助与剩余错误来源的讨论才有了可靠前提。

### 5.4 `fallback_text` 对 top-1 的劫持问题

在 benchmark 修复后，系统性能虽然显著提升，但 top-1 仍存在一批错误。`outputs/eval/docvqa_val_unique_docpage_100_top1_error_analysis_stronger_v2.md` 表明，这些错误并不主要来自真实 OCR 块排序失败，而是大量来自 synthetic `fallback_text` 对词面重叠的劫持：在优化前，`99/100` 的 top-1 结果来自 `fallback_text`，且 `99/100` 来自 `parser=fallback`，只有 `1/100` 来自真实 MinerU OCR 块。

这一现象揭示了一个非常关键的问题：如果索引中同时存在真实页面块与“问题回声块”，稀疏检索就可能优先匹配后者，从而让系统看似在检索，实则仍在“记住问题”。对此，项目在 `src/models/retrieval.py` 中对 `stronger` profile 做出调整，在稀疏预排序时先过滤 stopwords，减少整句题干回声对 lexical overlap 的支配作用。该改动使 `Hit@5` 从 `0.93` 提升到 `0.95`，`Precision@5` 从 `0.19` 提升到 `0.196`，`Citation@1` 从 `0.89` 提升到 `0.90`。更重要的是，它把优化方向重新拉回到了“页内真实证据排序”本身。

### 5.5 视觉稠密融合的负结果

在原始 benchmark 上，`exp5_densefusion_w045` 曾将 `Citation@1` 从 `0.12` 提升到 `0.13`，因此一度被视为较有前景的方向。然而，在 `unique-docpage-100 rebuilt` 上，对应的 `exp5_rebuilt` 并未超过文本优先主线，反而在 top-1 citation 上略弱于 `stronger_qia_v6`。

这一负结果的价值不应被低估。它说明视觉稠密融合并非天然优于文本优先检索。对于布局相近、模板重复、页面视觉元素高度相似的文档，视觉描述符容易把“相似页面”提前，而不一定把“正确证据页面”提前。因此，“把视觉信息尽早前置到检索阶段”在当前项目条件下并不是稳妥策略。

### 5.6 生成阶段视觉辅助的门控优化

当检索阶段视觉融合未能成为主线后，项目转向生成阶段视觉辅助。初版全局视觉辅助虽然在个别样本上改善了 `logo` 与 `title` 类问题，但整体 `ANLS` 与 `Token-F1` 出现下降，说明视觉信息并非越多越好，若缺乏题型约束，反而可能干扰稳定的文本 grounding。

基于 `outputs/eval/docvqa_val_unique_docpage_100_visual_assist_experiment.md` 的对比记录，项目最终引入门控策略，仅对 `logo`、`pack text`、`heading`、`title`、`page number` 与短手写文本问题启用视觉辅助，并对 `chart axis`、数字表格题和一般抽象语义题禁用该机制。该策略最终形成 `stronger_qia_visualassist_gated_v2`，并成为当前最稳健的端到端路线。这一结果说明，视觉信息的最佳系统角色不是“无差别增强”，而是“在文本证据已稳定时，对特定局部识别任务进行定向补偿”。

这一结论在科研叙事上同样十分关键。它意味着本文并未简单否定视觉信息的价值，而是通过一系列负结果与对照实验，完成了对视觉信息“系统角色”的重新界定。换言之，项目后期的贡献不是证明“视觉无用”，而是证明“视觉何时有用、为何在此时有用、以及为何不应在彼时强行介入”。

## 6 实验结果与分析

### 6.1 原始 `val-100` 上的历史结果

原始历史 benchmark 上的主要结果如下：


| line                    | type      | Hit@5 | Precision@5 | Citation@1 |   EM |     ANLS | Token-F1 |
| ----------------------- | --------- | ----: | ----------: | ---------: | ---: | -------: | -------: |
| `promptfix_v3`          | retrieval |  0.49 |       0.114 |       0.12 |    - |        - |        - |
| `exp5_densefusion_w045` | retrieval |  0.51 |       0.118 |       0.13 |    - |        - |        - |
| `promptfix_v3`          | rag       |  0.49 |       0.114 |       0.12 | 0.78 | 0.874770 | 0.897306 |
| `exp5_densefusion_w045` | rag       |  0.51 |       0.118 |       0.13 | 0.77 | 0.878745 | 0.898544 |

从表面上看，视觉稠密融合似乎略优于文本基线；但结合后续误差分析可以发现，这一差异并不足以支撑强结论。旧 benchmark 的主要作用更像是暴露问题，而不是提供最终架构判断。

### 6.2 修复 benchmark 后的检索结果

修复后主 benchmark 上的检索结果如下：


| line               | Hit@5 | Precision@5 | Citation@1 |
| ------------------ | ----: | ----------: | ---------: |
| `stronger_rebuilt` |  0.93 |        0.19 |       0.89 |
| `stronger_qia_v6`  |  0.95 |       0.196 |       0.91 |
| `exp5_rebuilt`     |  0.93 |        0.19 |       0.88 |

这一结果表明，原始 `val-100` 上的长期平台期并不能简单解释为“系统不会检索”。一旦 benchmark 去除了重复页污染，并以同一 manifest 重建检索语料，文本优先主线的能力便得到了更真实的呈现。尤其值得注意的是，修复 benchmark 后的 `stronger_qia_v6` 反而优于曾在历史实验中更被看好的视觉融合线，这直接支持了本文关于“文本优先主证据路径”的判断。

### 6.3 修复 benchmark 后的端到端结果

修复后主 benchmark 上的端到端 RAG 结果如下：


| line                                 | Hit@5 | Citation@1 |   EM |     ANLS | Answer-Contains | Token-F1 |
| ------------------------------------ | ----: | ---------: | ---: | -------: | --------------: | -------: |
| `stronger_qia_base`                  |  0.95 |       0.90 | 0.85 | 0.949966 |            0.92 | 0.936036 |
| `stronger_qia_visualassist_gated_v2` |  0.95 |       0.90 | 0.87 | 0.962800 |            0.94 | 0.951780 |

该结果具有三层含义。第一，加入门控视觉辅助后，检索与引用指标未下降，说明该机制没有破坏 grounding 稳定性。第二，`EM`、`ANLS` 与 `Token-F1` 同时提升，说明视觉辅助确实帮助模型更好处理局部短文本、标题与页面短语识别问题。第三，这一提升是在文本证据主线已经稳定的基础上获得的，因此“视觉信息更适合作为生成阶段补偿”这一结论具有更强可信度。

### 6.4 对研究问题的回答

针对第 1 节提出的三个研究问题，本文的回答如下：

1. 对于 RQ1，原始 `val-100` 上的平台期并非单纯模型能力不足，其主导因素是 benchmark 中大量重复页带来的 sample-level 偏差。
2. 对于 RQ2，在当前项目条件下，视觉信息作为检索阶段稠密融合信号的收益并不稳定，而作为生成阶段的门控辅助时更能稳定提升回答质量。
3. 对于 RQ3，在修复 benchmark 后，系统剩余错误已不再主要来自普通排序噪声，而更多来自 deictic/generic query 的 grounding 不足，以及更深层的 OCR、实体识别与版面理解缺口。

将三项回答合并来看，本文形成了一条相对完整的研究判断链：旧 benchmark 上的“平台期”首先是评测定义问题，其次才是检索与表示问题；在任务定义被修复之后，文本优先主线能够较稳定地承担证据排序职责；而当主证据路径趋于稳定后，视觉信息才以门控辅助的方式表现出正收益。也就是说，本文最终的研究结论并非来自某一次单点提分，而是来自问题定义、系统结构与实验结果三者逐步对齐之后的综合判断。

## 7 讨论

### 7.1 相比开题设想的真实变化

开题报告中的系统设想更接近“结构化切分 + 双路径 RAG + 晚融合”的典型多模态方案。结题阶段表面上仍保留了这一框架，但其内部逻辑已经发生重要变化：系统不再默认认为“多模态越早介入越好”，而是通过实验逐步建立了更细化的角色分配，即文本路径负责主证据排序，视觉路径负责局部识别补偿，benchmark 修复负责保障实验可解释性。

这种变化说明，项目的成熟并不只体现在功能变多或分数升高，更体现在系统主次关系的认识更加清晰。相比“证明某个模块有用”，本文更重要的贡献在于澄清“该模块在什么条件下、以什么方式有用”。

### 7.2 本项目最值得保留的研究经验

从研究方法角度看，本项目最值得保留的经验主要有三点。

第一，benchmark 并不是中性的。若评测对象与真实任务对象不一致，分数既可能夸大问题，也可能掩盖真实进步。第二，多模态信息不是天然正增益。其有效性取决于介入时机、题型条件与系统中的证据角色。第三，工程闭环与科学诊断同等重要。只有当 schema、API、citation、评测脚本与误差分析被统一到同一框架内，实验现象才真正具备可解释性。

如果进一步概括，这三条经验实际上共同指向一个更一般的研究原则：对于工程型多模态系统，真正高质量的结题工作不应仅展示“做成了什么”，更应解释“为何这样做、为何此前的做法不够好、以及结论为何值得相信”。本文的主要价值，正是在课程项目范围内尽可能完成了这一层解释任务。

## 8 局限性与后续工作

尽管当前系统已形成较完整的离线评测闭环，但仍存在以下局限。

1. 当前主要结论建立在 DocVQA 风格数据上，`unique-docpage-100` 虽比旧基准更合理，但仍属于固定切片，外部泛化能力仍需在更多文档域上验证。
2. 当前检索优化仍包含较强启发式成分，虽然在课程项目中有效，但未必能无缝推广到更大规模开放域语料。
3. 当前 benchmark 主要衡量页级检索与页级 grounding，而非更细粒度的区域级定位；页面正确不等于区域必然正确。
4. 在线文件上传、增量解析、实时建索引与长期 workspace 管理尚未成为正式实验对象，系统当前更接近研究原型而非完整产品。

据此，后续工作可沿三条路线展开：

1. 推进在线文档接入与增量索引，使系统从离线 benchmark 原型进一步走向真实交互应用。
2. 在现有 query-aware rerank 的基础上引入更学习化的页级或区域级 reranker，提升 generic/deictic query 的 grounding 能力。
3. 结合不确定性估计进一步细化视觉辅助门控策略，仅在模型对局部视觉短文本存在明显识别困难时触发视觉补偿。

## 9 结论

本文围绕课程文档场景下的多模态问答任务，完成了一套以块优先结构、可追溯引用与 benchmark 驱动诊断为核心的多模态 RAG 系统实现与实验分析。与开题阶段相比，结题阶段最重要的进展并不只是“做出了一个能运行的系统”，而是通过一系列失败分析、评测修复与实现重构，澄清了系统中真正的主要矛盾。

研究结果表明，原始 `DocVQA val first-100` 上看似持久的性能平台期，在很大程度上来自 benchmark 的重复页污染；在构造 `unique-docpage-100` 并重建对应索引语料后，文本优先检索能力得到了更真实的反映。同时，视觉信息虽然对页面理解不可或缺，但其最有效的系统角色并不是默认前置到检索排序，而是在文本证据已经稳定的前提下，以题型敏感的方式辅助生成模型完成局部视觉辨识。

因此，本文最终给出的并非一个泛泛的“多模态优于纯文本”的结论，而是一个条件更明确、也更具工程与研究价值的判断：对于课程规模的多模态文档问答系统，评测对象与索引对象的一致性、文本优先的稳定检索主线，以及题型敏感的门控视觉辅助机制，比简单堆叠更多多模态融合模块，更能带来可信、可解释且可复现的系统提升。
