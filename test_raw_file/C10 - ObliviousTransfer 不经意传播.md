## 一、概念
### 1、为什么需要OT
现实多方安全通信中，经常遇到两难困境：用户B想从用户A手中获取某条信息，但出于隐私考虑，B不希望A知道他拿走了哪一条。
- 敏感频道付费：用户付费观看视频，用户不希望平台记录并暴露自己观看了哪些敏感或特定政治/医疗频道 。
- 敏感商品订购：消费者在网上订购特殊药品，但不希望电商平台或物流发起者明确追踪其购买的特定药品种类。
### 2、OT协议经典形式
OT协议是一个由发送方A和接收方B参与的双方通信协议。根据单词传输中可选数据的总量和选择量，OT主要分为：
- **$1\text{-out-of-}2$ 不经意传输（$OT_2^1$）**：发送方 A 拥有两条消息 $m_0$ 与 $m_1$ 。 协议执行后，接收方 B 只能得到其中指定的一条 。
- **$1\text{-out-of-}n$ 不经意传输（$OT_n^1$）**：A 有 $n$ 条消息 $m_1, m_2, \dots, m_n$ ，B 最终只能且确切得到其中的某一条 $m_i$ 。
- **$k\text{-out-of-}n$ 不经意传输（$OT_n^k$）**：A 有 $n$ 条消息 ，B 最终得到其中的 $k$ 个消息（$k < n$） 。
### 3、3大安全属性
一个合格的 OT 协议必须同时满足以下三个性质 ：
1. **正确性（Correctness）**：如果双方都严格按照协议步骤执行，B 必然能顺利解密并得到他选择的那条消息 。
2. **不经意性/接收者隐私性（Receiver Privacy）**：发送方 A 无法通过通信数据（传送副本）区分 B 到底选了哪条消息 。 对 A 而言，B 的选择是不可区分的 。
3. **安全性/发送者隐私性（Sender Privacy）**：接收方 B **绝对无法**得到他选择之外的任何消息 。 其余消息的密文对 B 来说等同于随机乱码，不可解码 。
## 二、经典OT传输方案
### 1、基于对称/公钥加密的经典$OT_2^1$ 方案
使用盲化思想，使发送方无法分辨哪个是真实地会话密钥。
- 初始化：发送方 A 生成一套公钥系统 $(\mathcal{E}, \mathcal{D})$，并随机选择两个密文空间中的随机数 $c_0, c_1$ 。 A 将公钥以及 $c_0, c_1$ 发送给 B。
- 选择与盲化：B 拥有选择比特 $r \in \{0, 1\}$ 和一个随机生成的对称会话密钥 $k$ 。 B 用公钥加密 $k$，并用 A 给的 $c_r$ 进行**异或盲化** ：$$c = \mathcal{E}(k) \oplus c_r$$B 将 $c$ 发回给 A 。
- 解密候选密钥：A 收到 $c$ 后，由于不知道 B 选的是 $c_0$ 还是 $c_1$，A 只好把两种可能都算一遍 ：$$k_0' = \mathcal{D}(c \oplus c_0), \quad k_1' = \mathcal{D}(c \oplus c_1)$$两个结果中，必有一个是真实的B的会话密钥k。A无法分辨。
- 加密传输：A 用 $k_0'$ 加密 $m_0$，用 $k_1'$ 加密 $m_1$，全部发给 B ：$$C_0 = \mathcal{E}_{k_0'}(m_0), \quad C_1 = \mathcal{E}_{k_1'}(m_1)$$
- 解密：B用自己持有的真实会话密钥k去解密，因为他有真实的k，所以能读出自己选的那份消息，而另一份解出来是乱码 。
### 2、基于离散对数的$OT_n^1$ 方案
当消息数量变为n时，可以利用循环群上的离散对数难题DLP来实现。
- 系统参数：$G$ 是一个 $q$ 阶循环群，$g, h$ 是 $G$ 的两个生成元，且离散对数 $\log_g h$ **严格保密** 。 A 持有消息 $m_1, m_2, \dots, m_n \in G$ 。
- B发起选择：B 想要第 $a$ 个消息（$1 \le a \le n$），随机选择 $r \in \mathbb{Z}_q$，计算并发送 ：$$y = g^r h^a$$
- A加密响应：A 收到 $y$ 后，对**每一个** $i = 1, 2, \dots, n$，都选一个独立的随机数 $k_i \in \mathbb{Z}_q$ ，计算并发送元组 ：$$c_i = \left(g^{k_i}, \, m_i \cdot \left(\frac{y}{h^i}\right)^{k_i}\right)$$
- B解密$c_a$：B 只关注他要的第 $a$ 个密文 $c_a = (u, v) = \left(g^{k_a}, m_a \cdot (y/h^a)^{k_a}\right)$ 。 B 计算 ：$m_a = \frac{v}{u^r}$
### 3、基础OT之间转换关系
- **基于 $OT_2^1$ 构造 $OT_n^1$**：可以通过调用 $\log_2 n$ 次 $OT_2^1$ 协议来实现 。
    - _原理简述_：将消息的索引 $I$ 编码为长度为 $l = \log_2 n$ 的二进制比特串 $I_1I_2\dots I_l$ 。 通过伪随机函数 $F_K$ 和 $\log_2 n$ 次对密钥对的选择，最终组装解密 。
- **基于 $OT_n^1$ 构造 $OT_n^k$**：最朴素的方法是将 $OT_n^1$ 独立地运行 $k$ 次 。 但缺点显而易见：会带来 $k$ 倍的计算与通信负担 。
## 三、OT扩展协议 OTE/IKNP
### 1、OTE
传统OT协议严重依赖公钥加密（如大数分解、离散对数），计算开销极大。如果在面对海量数据时直接调用百万次公钥OT，算力会直接崩溃。
**OTE 的基本思想**：**“以小博大”** 。 我们只需要消耗极少量的（例如 $k=128$ 个）基础公钥 $OT_2^1$ ，就能通过高效的**对称密码（哈希、异或）**，批量扩展生成任意大数量（$m \gg k$ 个）独立的 $OT_2^1$ 。
### 2、IKNP-OT
#### 输入准备
- **发送方 S**：持有 $m$ 对长度为 $l$ 的比特消息 $\{(x_{j,0}, x_{j,1})\}_{j=1}^m$ 。
- **接收方 R**：持有 $m$ 个选择比特 $r = (r_1, r_2, \dots, r_m)^T$ 。
- **安全参数**：$k$（通常为 128）
#### 具体步骤
- **向量与矩阵准备**：
    - 发送方 S 随机选择一个 $k$ 维的随机比特**行向量** $s = (s_1, s_2, \dots, s_k) \in \{0, 1\}^k$ 。
    - 接收方 R 随机选择一个 $m \times k$ 的比特**矩阵** $T = (t^1, t^2, \dots, t^k)$，其中 $t^i$ 为列向量 。
- **身份对调，反向执行基础 OT**：
    - 双方调用 $k$ 次基础 $OT_2^1$ 协议 。
    - **注意！** 此时 **S 充当接收方**，其选择输入为 $s_i$ ； **R 充当发送方**，其准备的两个传输消息为 $(t^i, \, t^i \oplus r)$ 。
- **矩阵组装**：
    - 执行完 $k$ 次基础 OT 后，S 拿到了 $k$ 个长度为 $m$ 的列向量 $q^i$ 。 S 把它们拼成一个 $m \times k$ 的矩阵 $Q = (q^1, q^2, \dots, q^k)$ 。
    - 根据 OT 的性质，必然满足：当 $s_i=0$ 时，$q^i = t^i$； 当 $s_i=1$ 时，$q^i = t^i \oplus r$ 。 统一写成：$$q^i = t^i \oplus (r \cdot s_i)$$
- **行视角加密与解密**
    - 如果我们把矩阵 $Q$ 和 $T$ 扭转过来，看它们的**行向量** $q_j$ 和 $t_j$（第 $j$ 行，$1 \le j \le m$） ，根据上述列关系，行向量必然满足：        $$q_j = t_j \oplus (r_j \cdot s)$$
    - **S 加密响应**：S 利用每一行 $q_j$ 和哈希函数 $H$，对 $m$ 对消息进行对称加密发给 R ：    $$y_{j,0} = x_{j,0} \oplus H(j, q_j), \quad y_{j,1} = x_{j,1} \oplus H(j, q_j \oplus s)$$
    - **R 解密**：R 针对每一行 $j$，计算并输出 ：$$z_j = y_{j, r_j} \oplus H(j, t_j)$$
- 正确性论证：为什么 R 算出来的 $z_j$ 刚好等于他想要的 $x_{j, r_j}$ ？ 我们分情况讨论 $r_j$ 的取值 ：
	- **情况 1：若 B 的选择比特 $r_j = 0$** 根据行公式：$q_j = t_j \oplus (0 \cdot s) = t_j$ 。 此时 S 加密所用的密钥为 $H(j, q_j) = H(j, t_j)$ 。 则 R 恢复的数据：$z_j = y_{j,0} \oplus H(j, t_j) = x_{j,0} \oplus H(j, q_j) \oplus H(j, t_j) = x_{j,0}$ 。
	- **情况 2：若 B 的选择比特 $r_j = 1$** 根据行公式：$q_j = t_j \oplus (1 \cdot s) = t_j \oplus s$ 。 两边同时异或 $s$，得到 $q_j \oplus s = t_j$。 此时 S 加密所用的密钥为 $H(j, q_j \oplus s) = H(j, t_j)$ 。 则 R 恢复的数据：$z_j = y_{j,1} \oplus H(j, t_j) = x_{j,1} \oplus H(j, q_j \oplus s) \oplus H(j, t_j) = x_{j,1}$ 。
## 四、不经意的基于签名的电子信封 OSBE
### 1、概念
OSBE时一种特殊的条件解密技术。
- 机制：发送者 A 想发送一个加密秘密消息 $SM$ 给接收者 B，但只有当 B 拥有**可信第三方（CA）对特定消息 $m$ 的合法签名 $\sigma(m)$** 时，B 才能打开信封解密出 $SM$ 。
- 不经意性：即使B无法打开信封（没有合法签名），A也无法察觉B到底有没有打开成功。
### 2、基于Schnorr签名的OSBE
准备工作（Schnorr 签名复习）
- 系统公开参数：大素数 $p, q$，生成元 $g$，私钥 $x$，公钥 $y = g^x \pmod p$ 。
- CA 对属性消息 $m$ 签发 Schnorr 签名 $(e, s)$ ：
    - 选随机数 $k$ ，计算 $r = g^k \pmod p$
    - $e = h(m \,||\, r)$ ，$s = (k - xe) \pmod q$
    - 验证等式是否成立：$e = h(m \,||\, g^s y^e \pmod p)$ 。
协议交互流（A 与 B 之间）
1. **B 向 A 提交盲化凭证**：
    - **合法接收者 $B_1$**（拥有签名 $e, s$）：随机选 $t \in \mathbb{Z}_q$ ，计算并发送 $(r', s')$：$$r' = g^s y^e \pmod p, \quad s' = (t + s) \pmod q$$
    - **非法接收者 $B_2$**（没有签名）：只能胡乱选择随机数 $k', t' \in \mathbb{Z}_q$ ，计算并发送伪造的：$$r' = g^{k'} \pmod p, \quad s' = t' \pmod q$$
2. **A 构造信封并响应**：
    - A 收到 $(r', s')$ 后 ，随机选择 $\tilde{t} \in \mathbb{Z}_q$ ，计算：$$u = \left(g^{s'} y^{h(m || r')} \cdot (r')^{-1}\right)^{\tilde{t}} \pmod p, \quad v = g^{\tilde{t}} \pmod p$$
    - A 用安全哈希 $\overline{h}(u)$ 作为对称密钥加密秘密消息 $SM$ ：$C = \mathcal{E}_{\overline{h}(u)}(SM)$ 。
    - A 将 $(v, C)$ 发送给 B 。
3. **B 尝试打开信封**：
    - B 收到后，统一利用自己手里的随机数 $t$ 尝试解密，计算候选的 $u' = v^t \pmod p$ 。
